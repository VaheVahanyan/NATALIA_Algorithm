# Обработка команд CAN в ПО НА NATALIA

## 1. Назначение документа

Настоящий документ описывает алгоритм обработки команд, принимаемых штатным ПО
платы управления гамма-спектрометра НА NATALIA по интерфейсу CAN.

Документ определяет:
- перечень команд управления (КУ);
- перечень команд телеметрии (КТ);
- допустимость команд в различных режимах работы;
- правила валидации входных данных;
- действия, выполняемые при приеме команды;
- состав ответных сообщений;
- правила обработки ошибочных и недопустимых команд.

Документ используется совместно с:
- основным описанием алгоритма;
- таблицей состояний и переходов;
- документом по аварийной и защитной логике;
- документами по отдельным режимам работы.

## 2. Общие правила обработки CAN-команд

### 2.1 Общие положения

ПО должно принимать сообщения по интерфейсу CAN в формате, определенном протоколом UniCAN.

В алгоритме выделяются два класса входных сообщений:
- команды управления (КУ);
- команды телеметрии (КТ).

### 2.2 Общие правила обработки КУ

Для каждой корректно принятой КУ ПО должно:
1. распознать `MSG_ID`;
2. проверить допустимость команды в текущем режиме;
3. проверить корректность содержимого команды;
4. сформировать решение о принятии или отклонении команды;
5. выдать ТС «Квитанция» не позднее установленного времени;
6. при успешном принятии выполнить действие, предусмотренное командой.

### 2.3 Общие правила обработки КТ

Для каждой корректно принятой КТ ПО должно:
1. распознать `MSG_ID`;
2. проверить допустимость команды в текущем режиме;
3. проверить корректность содержимого команды;
4. при допустимости сохранить/использовать переданные данные согласно алгоритму;
5. не выдавать ТС «Квитанция».

### 2.4 Общие правила обработки ошибок

Если входная команда:
- содержит недопустимый `MSG_ID`;
- содержит недопустимое содержимое;
- не разрешена в текущем режиме,

то:
- КУ должна быть отклонена с выдачей ТС «Квитанция» с кодом ошибки;
- КТ должна быть проигнорирована;
- изменение режима работы и выполнение действий не производится.

## 4. Коды ошибок квитанции

| Код | Наименование | Смысл |
|---|---|---|
| 0 | OK | Команда принята к исполнению |
| 1 | ERR_MSG_ID | Недопустимый MSG_ID |
| 2 | ERR_CONTENT | Недопустимое содержимое команды |
| 3 | ERR_MODE | Команда недопустима в текущем режиме |
| 4 | ERR_OTHER | Иная ошибка |

Признак исполнения КУ:
- `0` — команда будет выполнена;
- `1` — команда отклонена.

## 7. Общий шаблон обработки КУ

Для любой КУ алгоритм обработки имеет следующий общий вид:

1. Принять сообщение от CAN/UniCAN драйвера.
2. Проверить корректность длины и `MSG_ID`.
3. Идентифицировать тип КУ.
4. Проверить, разрешена ли команда в текущем режиме.
5. Проверить корректность полезной нагрузки команды.
6. При ошибке:
   - сформировать ТС «Квитанция» с кодом ошибки;
   - завершить обработку.
7. При успехе:
   - сформировать ТС «Квитанция» с признаком принятия;
   - инициировать соответствующее действие;
   - при необходимости изменить режим;
   - при необходимости сформировать ТС «Статус».

---

## 8. Общий шаблон обработки КТ

Для любой КТ алгоритм обработки имеет следующий общий вид:

1. Принять сообщение от CAN/UniCAN драйвера.
2. Проверить корректность длины и `MSG_ID`.
3. Идентифицировать тип КТ.
4. Проверить, разрешена ли команда в текущем режиме.
5. Проверить корректность полезной нагрузки команды.
6. При ошибке:
   - проигнорировать команду;
   - завершить обработку.
7. При успехе:
   - сохранить/обработать полученные данные;
   - при необходимости записать научный формат в NAND;
   - режим работы не изменять.

## 9. Подробная обработка команд управления

---

### 9.1 CMD_TELEM_REQ — Запрос телеметрии

**MSG_ID:** `0000h`  
**Тип:** короткое сообщение  
**Ответ:** ТС «Квитанция», затем ТС «Телеметрия»  
**Разрешена в режимах:** `DUTY`, `OBSERVE`, `ALARM`

#### Входные поля
- байты 2-7: `AAh`

#### Проверки
- корректный `MSG_ID`;
- корректная длина;
- содержимое равно ожидаемому шаблону;
- команда разрешена в текущем режиме.

#### Действия при успехе
1. Сформировать ТС «Квитанция» со статусом `OK`.
2. Сформировать актуальное ТС «Телеметрия`.
3. Передать ТС «Телеметрия» в БВС КА.

#### Действия при ошибке
- при недопустимом режиме: квитанция `ERR_MODE`;
- при некорректном содержимом: квитанция `ERR_CONTENT`;
- режим не изменяется.

#### Изменение состояния
Не изменяет верхний режим.

---

#### Изменение состояния

---

### 9.4 CMD_OBSERVE_START — Включение режима наблюдений

**MSG_ID:** `0003h`  
**Тип:** короткое сообщение  
**Ответ:** ТС «Квитанция`, затем ТС «Статус`  
**Разрешена в режимах:** `DUTY`

#### Входные поля
- аппаратная конфигурация;
- параметры режима наблюдений;
- конфигурация триггера.

#### Проверки
- корректность выбора банка NAND;
- допустимость битов конфигурации;
- непротиворечивость параметров ПЭД;
- отсутствие признака переполнения выбранного банка;
- допустимость параметров форматов «События», «Спектр-1/2».

#### Действия при успехе
1. Сформировать ТС «Квитанция» со статусом `OK`.
2. Подготовить питание выбранного банка NAND.
3. При необходимости включить ПЭД.
4. Выполнить тест/проверку готовности ПЭД.
5. Записать текущую конфигурацию наблюдений.
6. Инкрементировать идентификатор сеанса наблюдений.
7. Перейти в режим `OBSERVE`.
8. Сформировать ТС «Статус».

#### Действия при ошибке
- переполнен выбранный банк: `ERR_CONTENT` или `ERR_OTHER` по принятому правилу;
- недопустимый режим: `ERR_MODE`;
- некорректная конфигурация: `ERR_CONTENT`.

#### Изменение состояния
`DUTY -> OBSERVE`

---

### 9.5 CMD_OBSERVE_CTRL — Управление режимом наблюдений

**MSG_ID:** `0004h`  
**Тип:** короткое сообщение  
**Ответ:** ТС «Квитанция`  
**Разрешена в режимах:** `OBSERVE`

#### Входные поля
- аппаратная конфигурация;
- параметры режима наблюдений;
- конфигурация триггера.

#### Проверки
- корректность битов управления ПЭД;
- допустимость комбинации форматов;
- допустимость значений параметров.

#### Действия при успехе
1. Сформировать ТС «Квитанция» со статусом `OK`.
2. Изменить настройки режима наблюдений.
3. Обновить конфигурацию триггера и ПЭД.
4. Зафиксировать смену номера режима наблюдений.
5. Запланировать формирование формата «Телеметрия» на следующем тике 1 Гц.

#### Изменение состояния
Остаётся в `OBSERVE`.

### 9.6 CMD_DUTY — Дежурный режим

**MSG_ID:** `0005h`  
**Тип:** короткое сообщение  
**Ответ:** ТС «Квитанция`, затем ТС «Статус`  
**Разрешена в режимах:** `DUTY`, `ERASE`, `TEST`, `OBSERVE`, `DUMP`

#### Входные поля
- аппаратная конфигурация дежурного режима.

#### Проверки
- корректность выбора банка;
- допустимость конфигурации питания;
- допустимость команды в текущем режиме.

#### Действия при успехе
1. Сформировать ТС «Квитанция».
2. Завершить или прервать текущую целевую операцию по правилам режима.
3. Установить аппаратную конфигурацию дежурного режима.
4. Перейти в `DUTY`.
5. Сформировать ТС «Статус`.

#### Изменение состояния
- `ERASE -> DUTY`
- `TEST -> DUTY`
- `OBSERVE -> DUTY`
- `DUMP -> DUTY`
- `DUTY -> DUTY`

### 9.8 CMD_SET_CFG — Задание настроек

**MSG_ID:** `0007h`  
**Тип:** длинное сообщение  
**Ответ:** ТС «Квитанция`  
**Разрешена в режимах:** `DUTY`, `ALARM`

#### Входные поля
- полный набор параметров конфигурации;
- маска аварий;
- параметры порогов;
- идентификатор сеанса и счётчики по управляющему слову.

#### Проверки
- корректность длины;
- допустимость численных диапазонов;
- корректность управляющего слова записи;
- допустимость команды в текущем режиме.

#### Действия при успехе
1. Сформировать ТС «Квитанция».
2. Обновить текущую конфигурацию.
3. Выполнить запись в основной и резервный MRAM.
4. При необходимости пересчитать masked alarm.
5. При необходимости инициировать выход из `ALARM` после отдельной проверки.

#### Изменение состояния
Не изменяет верхний режим напрямую.

### 9.13 CMD_RESET_ALARM — Сброс аварийного статуса

**MSG_ID:** `000Ch`  
**Тип:** короткое сообщение  
**Ответ:** ТС «Квитанция`  
**Разрешена в режимах:** `ALARM`

#### Входные поля
- байты 2-7: `AAh`

#### Проверки
- корректность содержимого;
- допустимость команды в текущем режиме.

#### Действия при успехе
1. Сформировать ТС «Квитанция`.
2. Обнулить параметр «Аварийный статус НА».
3. Пересчитать «Замаскированный аварийный статус НА».
4. Если masked alarm стал равен 0, дальнейший переход в `DUTY` выполняется общей логикой автомата.

#### Изменение состояния
Напрямую не переводит состояние, но может привести к `ALARM -> DUTY`.
