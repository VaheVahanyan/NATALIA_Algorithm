# Обработка команд CAN в ПО НА NATALIA

## 1. Назначение документа

Настоящий документ описывает алгоритм обработки команд, принимаемых штатным ПО платы управления гамма-спектрометра НА
NATALIA по интерфейсу CAN.

Документ определяет:

- перечень команд управления;
- перечень команд телеметрии;
- допустимость команд в различных режимах работы;
- правила проверки входных данных;
- действия, выполняемые при приеме команды;
- состав ответных сообщений;
- правила обработки ошибочных и недопустимых команд.

Документ используется совместно с:

- таблицей состояний и переходов;
- документами по отдельным режимам работы;
- документом допущений и открытых вопросов;
- диаграммами состояний Mermaid.

---

## 2. Общие правила обработки CAN-команд

### 2.1 Общие положения

ПО должно принимать сообщения по интерфейсу CAN1 в формате, определенном протоколом UniCAN.

В алгоритме выделяются два класса входных сообщений:

- команды управления;
- команды телеметрии.

### 2.2 Общие правила обработки КУ

Для каждой корректно принятой КУ ПО должно:

1. распознать `MSG_ID`;
2. проверить допустимость команды в текущем режиме;
3. проверить корректность содержимого команды;
4. сформировать решение о принятии или отклонении команды;
5. выдать ТС «Квитанция» в установленный протоколом срок;
6. при успешном принятии выполнить действие, предусмотренное командой.

### 2.3 Общие правила обработки КТ

Для каждой корректно принятой КТ ПО должно:

1. распознать `MSG_ID`;
2. проверить допустимость команды в текущем режиме;
3. проверить корректность содержимого команды;
4. при допустимости сохранить и использовать переданные данные в соответствии с алгоритмом;
5. не выдавать ТС «Квитанция».

### 2.4 Общие правила обработки ошибок

Если входная команда:

- содержит недопустимый `MSG_ID`;
- содержит недопустимое содержимое;
- не разрешена в текущем режиме,

то:

- КУ должна быть отклонена с выдачей ТС «Квитанция» с кодом ошибки;
- КТ должна быть проигнорирована;
- изменение режима работы и выполнение целевого действия не производится.

### 2.5 Автоматические ТС «Статус»

ТС «Статус» должно выдаваться автоматически при изменении параметров:

- «Режим работы НА»;
- «Идентификатор переполнения ППЗУ».

### 2.6 Уровень детализации

Настоящий документ описывает:

- допустимость команд;
- правила валидации;
- верхнеуровневые действия по командам;
- состав ответных сообщений.

Внутренние последовательности действий конкретных режимов описываются в соответствующих режимных документах.

## 4. Коды ошибок квитанции

| Код | Обозначение   | Смысл                                |
|-----|---------------|--------------------------------------|
| 0   | `OK`          | Команда принята к исполнению         |
| 1   | `ERR_MSG_ID`  | Недопустимый `MSG_ID`                |
| 2   | `ERR_CONTENT` | Недопустимое содержимое команды      |
| 3   | `ERR_MODE`    | Команда недопустима в текущем режиме |
| 4   | `ERR_OTHER`   | Иная ошибка                          |

Признак исполнения КУ:

- `0` — команда принята к исполнению;
- `1` — команда отклонена.

## 7. Общий шаблон обработки КУ

Для любой КУ используется следующая последовательность:

1. Принять сообщение от CAN/UniCAN-драйвера.
2. Проверить корректность длины и `MSG_ID`.
3. Идентифицировать тип КУ.
4. Проверить допустимость команды в текущем режиме.
5. Проверить корректность полезной нагрузки.
6. При ошибке:
    - сформировать ТС «Квитанция» с кодом ошибки;
    - завершить обработку.
7. При успехе:
    - сформировать ТС «Квитанция» со статусом принятия;
    - выполнить действия, предусмотренные командой;
    - при необходимости изменить режим;
    - при необходимости выдать ТС «Статус», ТС «Телеметрия» или иное предусмотренное сообщение.

---

## 8. Общий шаблон обработки КТ

Для любой КТ используется следующая последовательность:

1. Принять сообщение от CAN/UniCAN-драйвера.
2. Проверить корректность длины и `MSG_ID`.
3. Идентифицировать тип КТ.
4. Проверить допустимость команды в текущем режиме.
5. Проверить корректность полезной нагрузки.
6. При ошибке:
    - проигнорировать команду;
    - завершить обработку.
7. При успехе:
    - сохранить и обработать полученные данные;
    - при необходимости сформировать соответствующий формат НИ;
    - режим работы не изменять.

## 9. Подробная обработка команд управления

### 9.1 `CMD_TELEM_REQ` — Запрос телеметрии

**MSG_ID:** `0000h`  
**Тип:** короткое сообщение  
**Ответ:** ТС «Квитанция», затем ТС «Телеметрия»  
**Разрешена в режимах:** `DUTY`, `OBSERVE`, `ALARM`

#### Входные поля

- байты 2-7: `AAh`

#### Проверки

- корректный `MSG_ID`;
- корректная длина;
- содержимое соответствует ожидаемому шаблону;
- команда разрешена в текущем режиме.

#### Действия при успехе

1. Сформировать ТС «Квитанция» со статусом `OK`.
2. Сформировать актуальное ТС «Телеметрия`.
3. Передать ТС «Телеметрия» в БВС КА.

#### Действия при ошибке

- при недопустимом режиме: квитанция `ERR_MODE`;
- при некорректном содержимом: квитанция `ERR_CONTENT`.

#### Изменение состояния

Не изменяет текущий режим.

---

**Разрешена в режимах:** `DUTY`, `ERASE`, `TEST`, `OBSERVE`, `DUMP`, `ALARM`, `SHUTDOWN`

#### Входные поля

#### Проверки

- корректный `MSG_ID`;

3. Передать ТС «Статус» в БВС КА.

#### Изменение состояния

Не изменяет текущий режим.

---

### 9.3 `CMD_SET_TIME` — Предустановка времени

**MSG_ID:** `0002h`  
**Тип:** короткое сообщение  
**Ответ:** ТС «Квитанция`  
**Разрешена в режимах:** `DUTY`

#### Входные поля

- бортовое время;
- миллисекунды.

#### Проверки

- корректная длина;
- допустимый формат времени;
- команда разрешена в текущем режиме.

#### Действия при успехе

1. Сформировать ТС «Квитанция» со статусом `OK`.
2. Выполнить предустановку времени RTC.
3. Обновить связанные программные структуры, если это требуется.

#### Действия при ошибке

- при недопустимом режиме: квитанция `ERR_MODE`;
- при некорректном содержимом: квитанция `ERR_CONTENT`.

#### Изменение состояния

Не изменяет текущий режим.

---

### 9.4 `CMD_OBSERVE_START` — Включение режима наблюдений

**MSG_ID:** `0003h`  
**Тип:** короткое сообщение  
**Ответ:** ТС «Квитанция», затем ТС «Статус`  
**Разрешена в режимах:** `DUTY`

#### Входные поля

- аппаратная конфигурация;
- параметры режима наблюдений;
- конфигурация триггера.

#### Проверки

- корректная длина;
- корректность выбора банка NAND;
- допустимость битов аппаратной конфигурации;
- допустимость параметров форматов «События», «Спектр-1» и «Спектр-2`;
- допустимость конфигурации ПЭД;
- отсутствие блокирующего `MaskedAlarm`;
- выбранный банк допускает запуск наблюдений.

#### Действия при успехе

1. Сформировать ТС «Квитанция» со статусом `OK`.
2. Передать управление подалгоритму `OBSERVE`.
3. После перехода в `OBSERVE` сформировать ТС «Статус».

#### Действия при ошибке

- недопустимый режим: `ERR_MODE`;
- некорректная конфигурация: `ERR_CONTENT`;
- отказ запуска наблюдений: `ERR_OTHER`.

#### Изменение состояния

`DUTY -> OBSERVE`

---

### 9.5 `CMD_OBSERVE_CTRL` — Управление режимом наблюдений

**MSG_ID:** `0004h`  
**Тип:** короткое сообщение  
**Ответ:** ТС «Квитанция`  
**Разрешена в режимах:** `OBSERVE`

#### Входные поля

- аппаратная конфигурация;
- параметры режима наблюдений;
- конфигурация триггера.

#### Проверки

- корректная длина;
- корректность битов управления ПЭД;
- допустимость комбинации форматов;
- допустимость значений параметров;
- команда разрешена в текущем режиме.

#### Действия при успехе

1. Сформировать ТС «Квитанция» со статусом `OK`.
2. Обновить конфигурацию режима наблюдений.
3. Обновить конфигурацию ПЭД через `PED_REG`.
4. Инкрементировать номер режима наблюдений.
5. Запланировать формирование формата «Телеметрия» на следующем `RTC 1 Гц`.

#### Действия при ошибке

- недопустимый режим: `ERR_MODE`;
- некорректные параметры: `ERR_CONTENT`;
- внутренняя ошибка реконфигурации: `ERR_OTHER`.

#### Изменение состояния

Остается в `OBSERVE`.

---

### 9.6 `CMD_DUTY` — Дежурный режим

**MSG_ID:** `0005h`  
**Тип:** короткое сообщение  
**Ответ:** ТС «Квитанция», в ряде режимов затем ТС «Статус`  
**Разрешена в режимах:** `DUTY`, `ERASE`, `TEST`, `OBSERVE`, `DUMP`

#### Входные поля

- аппаратная конфигурация дежурного режима.

#### Проверки

- корректная длина;
- допустимость содержимого команды;
- команда разрешена в текущем режиме.

#### Действия при успехе в режиме `DUTY`

1. Сформировать ТС «Квитанция» со статусом `OK`.
2. Не изменять текущий режим.

#### Действия при успехе в режимах `ERASE`, `TEST`, `OBSERVE`, `DUMP`

1. Сформировать ТС «Квитанция» со статусом `OK`.
2. Выполнить завершение текущего режима по правилу соответствующего подалгоритма.
3. После перехода в `DUTY` сформировать ТС «Статус`.

#### Действия при ошибке

- недопустимый режим: `ERR_MODE`;
- некорректное содержимое: `ERR_CONTENT`.

#### Изменение состояния

- `ERASE -> DUTY`
- `TEST -> DUTY`
- `OBSERVE -> DUTY`
- `DUMP -> DUTY`
- `DUTY -> DUTY`

### 9.7 `CMD_DUMP` — Вывод данных

**MSG_ID:** `0006h`  
**Тип:** короткое сообщение  
**Ответ:** ТС «Квитанция`, затем ТС «Статус`  
**Разрешена в режимах:** `DUTY`

#### Входные поля

- аппаратная конфигурация;
- количество пакетов;
- тип вывода;
- интерфейс вывода.

#### Проверки

- корректная длина;
- корректность выбора банка NAND;
- допустимость числа пакетов;
- допустимость параметров вывода;
- наличие данных для вывода;
- допустимость интерфейса вывода;
- команда разрешена в текущем режиме.

#### Действия при успехе

1. Сформировать ТС «Квитанция» со статусом `OK`.
2. Передать управление подалгоритму `DUMP`.
3. После перехода в `DUMP` сформировать ТС «Статус`.

#### Действия при ошибке

- недопустимый режим: `ERR_MODE`;
- некорректное содержимое: `ERR_CONTENT`;
- невозможность запуска вывода: `ERR_OTHER`.

#### Изменение состояния

`DUTY -> DUMP`

#### Примечание

В алгоритмической логике используется только вывод данных по USB.

### 9.8 `CMD_SET_CFG` — Задание настроек

**MSG_ID:** `0007h`  
**Тип:** длинное сообщение  
**Ответ:** ТС «Квитанция`  
**Разрешена в режимах:** `DUTY`, `ALARM`

#### Входные поля

- полный набор параметров конфигурации;
- маска аварий;
- пороговые параметры;
- служебные поля команды.

#### Проверки

- корректная длина;
- допустимость диапазонов значений;
- корректность управляющих полей;
- допустимость маскирования аварийных признаков;
- команда разрешена в текущем режиме.

#### Действия при успехе

1. Сформировать ТС «Квитанция» со статусом `OK`.
2. Обновить подготавливаемую конфигурацию.
3. Выполнить запись конфигурации в первую копию MRAM.
4. Выдержать необходимый интервал завершения записи.
5. Выполнить запись конфигурации во вторую копию MRAM.
6. Сделать новую конфигурацию текущей рабочей конфигурацией.
7. Пересчитать `MaskedAlarm`.

#### Дополнительные действия в `DUTY`

- если после пересчета `MaskedAlarm == 0`, остаться в `DUTY`;
- если после пересчета `MaskedAlarm != 0`, перейти в `ALARM` и сформировать ТС «Статус`.

#### Дополнительные действия в `ALARM`

- если после пересчета `MaskedAlarm != 0`, остаться в `ALARM`;
- если после пересчета `MaskedAlarm == 0`, перейти в `DUTY` и сформировать ТС «Статус`.

#### Действия при ошибке

- недопустимый режим: `ERR_MODE`;
- некорректные параметры: `ERR_CONTENT`;
- ошибка записи конфигурации: `ERR_OTHER`.

#### Изменение состояния

- в `DUTY` может не изменять состояние либо приводить к `ALARM`;
- в `ALARM` может не изменять состояние либо приводить к `DUTY`.

### 9.9 `CMD_ERASE` — Стирание NAND

**MSG_ID:** `0008h`  
**Тип:** короткое сообщение  
**Ответ:** ТС «Квитанция`, затем ТС «Статус`  
**Разрешена в режимах:** `DUTY`

#### Входные поля

- идентификатор банка NAND;
- признак состояния питания банка после завершения.

#### Проверки

- корректная длина;
- корректность выбора банка;
- команда разрешена в текущем режиме.

#### Действия при успехе

1. Сформировать ТС «Квитанция» со статусом `OK`.
2. Передать управление подалгоритму `ERASE`.
3. После перехода в `ERASE` сформировать ТС «Статус`.

#### Действия при ошибке

- недопустимый режим: `ERR_MODE`;
- некорректное содержимое: `ERR_CONTENT`;
- невозможность запуска стирания: `ERR_OTHER`.

#### Изменение состояния

`DUTY -> ERASE`

### 9.11 `CMD_TEST_RESULT` — Запрос результатов теста NAND

**MSG_ID:** `000Ah`  
**Тип:** короткое сообщение  
**Ответ:** ТС «Квитанция`, затем ТС «Результаты теста ППЗУ`  
**Разрешена в режимах:** `DUTY`

#### Входные поля

- идентификатор банка NAND.

#### Проверки

- корректность выбора банка;

#### Действия при успехе

2. Считать сохраненный массив результатов теста.
3. Сформировать ТС «Результаты теста ППЗУ`.
4. Передать ТС в БВС КА.

#### Действия при ошибке

- некорректный банк: `ERR_CONTENT`;
- отсутствие валидного результата: `ERR_OTHER`.

#### Изменение состояния

Не изменяет текущий режим.

**MSG_ID:** `000Bh`  
**Тип:** короткое сообщение

#### Действия при ошибке

#### Изменение состояния

`{DUTY, ERASE, TEST, OBSERVE, DUMP, ALARM} -> SHUTDOWN`

### 9.13 `CMD_RESET_ALARM` — Сброс аварийного статуса

**MSG_ID:** `000Ch`  
**Тип:** короткое сообщение  
**Ответ:** ТС «Квитанция`  
**Разрешена в режимах:** `ALARM`

#### Входные поля

- байты 2-7: `AAh`

#### Проверки

- корректная длина;
- корректное содержимое;
- команда разрешена в текущем режиме.

#### Действия при успехе

1. Сформировать ТС «Квитанция» со статусом `OK`.
2. Обнулить `AlarmStatus`.
3. Пересчитать `MaskedAlarm`.
4. Если после пересчета `MaskedAlarm == 0`, выполнить выход из `ALARM` в `DUTY` и сформировать ТС «Статус`.

#### Действия при ошибке

- недопустимый режим: `ERR_MODE`;
- некорректное содержимое: `ERR_CONTENT`.

#### Изменение состояния

Обычно остается в `ALARM`, но может привести к `ALARM -> DUTY`.

#### Примечание

Если физическая причина аварии не устранена, соответствующие аварийные биты могут быть выставлены повторно следующим
циклом контроля.
