# Подалгоритм режима вывода данных ПО НА NATALIA

## 1. Назначение документа

Настоящий документ описывает подалгоритм работы штатного ПО в режиме вывода данных.

Документ определяет:
- условия входа в режим вывода данных;
- выбор источника данных и параметров вывода;
- последовательность чтения пакетов НИ из выбранного банка NAND;
- правила передачи данных во внешнюю систему;
- условия штатного завершения;
- условия досрочного завершения по команде;
- условия аварийного завершения;
- действия при выходе из режима.

Документ используется совместно с:
- основным описанием алгоритма;
- таблицей состояний и переходов;
- документом обработки команд CAN;
- документом аварийной и защитной логики;
- описанием форматов научной информации;
- техническими требованиями на ПУ;
- описанием интерфейсов МК.

---

## 2. Назначение режима вывода данных

Режим вывода данных предназначен для передачи накопленной научной информации
из выбранного банка NAND во внешнюю бортовую систему.

Согласно протоколу CAN:
- режим запускается только из `DUTY`;
- данные выводятся из заданного банка ППЗУ;
- вывод ведётся пакетами по `2 кБ`;
- после завершения вывода выполняется автоматический переход в `DUTY`;
- по команде задаются банк, интерфейс вывода, тип вывода и количество пакетов. :contentReference[oaicite:0]{index=0}

Согласно формату научной информации:
- основной единицей хранения является пакет НИ размером `2048 байт`;
- пакеты хранятся в NAND в хронологическом порядке. :contentReference[oaicite:1]{index=1}

Это удобно: размер пакета НИ и размер единицы вывода совпадают, что избавляет мир от ещё одной бессмысленной боли.

---

## 3. Условия входа в режим вывода данных

### 3.1 Внешнее условие входа
Вход выполняется по КУ:
- `CMD_DUMP` (`MSG_ID = 0006h`)

### 3.2 Допустимый исходный режим
Вход допускается только из:
- `DUTY`

### 3.3 Обязательные условия входа
Для успешного запуска должны выполняться:
- команда корректна по формату и содержимому;
- выбран корректный банк NAND;
- отсутствует ненулевой `MaskedAlarm`;
- в выбранном банке есть доступные данные, если этого требует тип вывода;
- интерфейс вывода допустим;
- сервис чтения NAND готов;
- сервис передачи данных готов.

### 3.4 Причины отказа во входе
Запуск режима отклоняется, если:
- команда пришла не из `DUTY`;
- указан недопустимый банк;
- указано недопустимое содержимое;
- нет данных для вывода;
- невозможно включить или использовать выбранный банк;
- активна авария;
- невозможно подготовить интерфейс передачи.

---

## 4. Входные параметры режима вывода данных

Режим конфигурируется КУ `CMD_DUMP`.

### 4.1 Аппаратная конфигурация
Команда задаёт:
- идентификатор используемого банка NAND:
  - `1` — `NAND1`
  - `2` — `NAND2`
- признак состояния питания банка после завершения вывода:
  - `0` — выключить
  - `1` — оставить включённым
- интерфейс вывода данных:
  - `0` — USB
  - `1` — CAN
- тип вывода данных:
  - `0` — вывести запрошенное количество пакетов
  - `1` — вывести все накопленные пакеты

### 4.2 Количество пакетов
Если тип вывода = `0`, команда содержит:
- число пакетов для передачи.

Если тип вывода = `1`, поле количества пакетов игнорируется. :contentReference[oaicite:2]{index=2}

---

## 5. Внутренние данные режима вывода

Во время вывода ПО должно поддерживать:

### 5.1 Постоянные на операцию
- выбранный банк NAND;
- выбранный интерфейс вывода;
- тип вывода;
- целевое число пакетов для передачи;
- требуемое состояние питания банка после завершения.

### 5.2 Динамические данные
- текущее подсостояние режима;
- индекс текущего пакета;
- число уже переданных пакетов;
- число оставшихся пакетов;
- текущий буфер пакета 2 кБ;
- флаг окончания вывода;
- флаг ошибки передачи;
- флаг ошибки чтения NAND.

### 5.3 Служебные данные
- число накопленных пакетов в выбранном банке;
- состояние питания выбранного банка;
- статус интерфейса вывода;
- статус `VBUS`, если используется USB.

Технические требования и описание интерфейсов прямо задают USB как интерфейс вывода
и сигнал `PU_USB_VBUS` как статус наличия питания на шине VBUS, что уже влияет на алгоритм,
а не только на схемотехнику. 

---

## 6. Общая последовательность входа в режим вывода

При успешном принятии `CMD_DUMP` ПО должно:

1. Выдать ТС «Квитанция».
2. Сохранить параметры команды.
3. Выбрать банк NAND.
4. Если питание банка выключено:
   - включить питание;
   - дождаться подтверждения, если используется логика `PSON`.
5. Определить число пакетов к передаче:
   - либо по полю команды,
   - либо по счётчику накопленных пакетов.
6. Подготовить интерфейс вывода.
7. Перевести верхний режим в `DUMP`.
8. Сформировать ТС «Статус`.
9. Начать цикл вывода пакетов.

---

## 7. Подсостояния внутри режима вывода

| Подсостояние | Обозначение | Назначение |
|---|---|---|
| Подготовка входа | DUMP_ENTER | Выбор банка, включение питания, расчёт объёма вывода |
| Подготовка интерфейса | DUMP_IF_PREP | Подготовка USB/CAN к передаче |
| Чтение пакета | DUMP_READ | Чтение очередного пакета из NAND |
| Передача пакета | DUMP_SEND | Передача очередного пакета наружу |
| Проверка завершения | DUMP_CHECK | Проверка, есть ли ещё пакеты для вывода |
| Штатное завершение | DUMP_FINISH_OK | Завершение вывода и выход |
| Завершение по команде | DUMP_FINISH_CMD | Выход по `CMD_DUTY` или `CMD_SHUTDOWN` |
| Завершение по аварии | DUMP_FINISH_ALARM | Выход при `MaskedAlarm != 0` или ошибке |

---

## 8. Основные источники событий в режиме вывода

### 8.1 Разрешённые КУ
В режиме вывода разрешены:
- `CMD_SHUTDOWN`
- `CMD_DUTY`
- `CMD_STATUS_REQ`

Остальные КУ запрещены. :contentReference[oaicite:4]{index=4}

### 8.2 КТ
Все КТ в режиме вывода игнорируются. :contentReference[oaicite:5]{index=5}

### 8.3 Внутренние события
Существенные события:
- подтверждение включения питания NAND;
- готовность интерфейса вывода;
- успешное чтение пакета;
- ошибка чтения пакета;
- успешная передача пакета;
- ошибка передачи пакета;
- отсутствие `VBUS` для USB;
- появление `MaskedAlarm != 0`.

---

## 9. Общая логика вывода данных

### 9.1 Базовая идея
Режим состоит из повторяющегося цикла:
1. прочитать очередной пакет НИ из NAND;
2. передать его наружу;
3. увеличить счётчик переданных пакетов;
4. решить, нужно ли продолжать.

### 9.2 Размерность вывода
Поскольку один пакет НИ равен `2048 байт`, он может быть естественной единицей чтения и передачи. :contentReference[oaicite:6]{index=6}

### 9.3 Приоритеты событий внутри режима
Рекомендуемый приоритет:
1. `MaskedAlarm != 0`
2. `CMD_SHUTDOWN`
3. ошибка чтения/передачи
4. завершение текущей передачи
5. `CMD_DUTY`
6. `CMD_STATUS_REQ`

---

## 10. Подготовка источника и интерфейса вывода

### 10.1 Подготовка банка NAND
ПО должно:
1. выбрать нужный банк;
2. обеспечить его питание;
3. проверить доступность чтения;
4. определить количество доступных пакетов.

### 10.2 Подготовка интерфейса вывода
ПО должно:
1. выбрать интерфейс (USB или CAN);
2. для USB проверить наличие `VBUS`, если это обязательно;
3. перевести сервис передачи в рабочее состояние;
4. зафиксировать готовность к передаче.

### 10.3 Ошибки подготовки
Ошибками считаются:
- недоступность банка;
- невозможность чтения;
- отсутствие `VBUS` при USB-выводе;
- невозможность инициализировать интерфейс;
- внутренняя ошибка сервиса передачи.

---

## 11. Этап чтения пакета

### 11.1 Назначение
На этом этапе ПО должно считать один пакет НИ объёмом `2048 байт` из выбранного банка NAND. :contentReference[oaicite:7]{index=7}

### 11.2 Действия алгоритма
ПО должно:
1. определить адрес/номер текущего пакета;
2. считать пакет в буфер;
3. проверить успешность чтения;
4. передать буфер на этап передачи.

### 11.3 Ошибки
Ошибками являются:
- отказ чтения;
- ошибка QSPI;
- потеря питания банка;
- нарушение целостности чтения, если алгоритм это проверяет.

---

## 12. Этап передачи пакета

### 12.1 Назначение
На этом этапе ПО должно передать считанный пакет через выбранный интерфейс.

### 12.2 Действия алгоритма
ПО должно:
1. инициировать передачу буфера;
2. дождаться подтверждения успешной передачи или завершения операции;
3. увеличить счётчик переданных пакетов;
4. перейти к проверке условия продолжения.

### 12.3 Ошибки
Ошибками являются:
- тайм-аут передачи;
- отказ интерфейса вывода;
- отсутствие `VBUS` для USB;
- ошибка протокольного обмена, если она трактуется как отказ операции.

Протокол CAN отдельно указывает, что авария USB включает исчезновение питания на `VBUS`
и тайм-аут при передаче данных. :contentReference[oaicite:8]{index=8}

---

## 13. Проверка завершения вывода

### 13.1 Условие завершения для типа вывода 0
Если тип вывода = “запрошенное количество пакетов”, режим завершается, когда:
- передано заданное количество пакетов.

### 13.2 Условие завершения для типа вывода 1
Если тип вывода = “все накопленные пакеты”, режим завершается, когда:
- переданы все накопленные пакеты выбранного банка.

### 13.3 Действие при незавершённости
Если пакеты ещё остались:
- алгоритм возвращается к чтению следующего пакета.

---

## 14. Штатное завершение вывода

### 14.1 Условие
Вывод считается завершённым штатно, если:
- передано всё требуемое количество пакетов;
- не возникло аварии;
- не возникло ошибки чтения или передачи.

### 14.2 Действия
При штатном завершении ПО должно:
1. завершить текущую передачу;
2. зафиксировать число переданных пакетов;
3. оставить питание банка включённым или отключить его по параметру команды;
4. перейти в `DUTY`;
5. сформировать ТС «Статус`.

Согласно CAN-протоколу, после завершения вывода данных должен быть переход в дежурный режим. :contentReference[oaicite:9]{index=9}

---

## 15. Досрочное завершение по команде

### 15.1 Команда `CMD_DUTY`
При получении `CMD_DUTY` ПО должно:
1. выдать квитанцию;
2. корректно завершить или остановить текущую передачу;
3. перейти в `DUTY`;
4. сформировать ТС «Статус`.

### 15.2 Команда `CMD_SHUTDOWN`
При получении `CMD_SHUTDOWN` ПО должно:
1. выдать квитанцию;
2. завершить или остановить передачу по принятому правилу;
3. при необходимости сохранить служебное состояние;
4. перейти в `SHUTDOWN`;
5. сформировать ТС «Статус`.

### 15.3 Открытый вопрос
Нужно отдельно зафиксировать, допускается ли:
- мгновенно оборвать передачу пакета;
- или обязательно завершать текущий пакет до выхода.

---

## 16. Завершение по аварии

### 16.1 Условие
Если в режиме вывода `MaskedAlarm != 0`, ПО должно перейти в `ALARM`. :contentReference[oaicite:10]{index=10}

### 16.2 Действия
При аварийном завершении ПО должно:
1. прекратить вывод данных;
2. отключить ПЭД;
3. отключить оба банка NAND;
4. зафиксировать переход в `ALARM`;
5. сформировать ТС «Статус`.

### 16.3 Типовые причины
Особенно важны:
- авария банка NAND;
- ошибка QSPI;
- авария USB;
- отсутствие `VBUS`;
- выход температур, токов, напряжений за допуск. 

---

## 17. Обработка разрешённых команд в режиме вывода

### 17.1 CMD_STATUS_REQ
При получении `CMD_STATUS_REQ`:
1. выдать квитанцию;
2. выдать ТС «Статус`;
3. режим не менять.

### 17.2 CMD_DUTY
При получении `CMD_DUTY`:
1. выдать квитанцию;
2. инициировать завершение/останов вывода;
3. перейти в `DUTY`;
4. выдать ТС «Статус`.

### 17.3 CMD_SHUTDOWN
При получении `CMD_SHUTDOWN`:
1. выдать квитанцию;
2. инициировать завершение/останов вывода;
3. перейти в `SHUTDOWN`;
4. выдать ТС «Статус`.

### 17.4 Остальные КУ
Остальные КУ:
- отклоняются;
- сопровождаются квитанцией с ошибкой;
- режим не меняют.

## 20. Открытые вопросы

В этом разделе фиксируются места, требующие уточнения:

- будет ли реально использоваться CAN как интерфейс вывода данных или только USB;
- как именно трактовать частичную ошибку передачи одного пакета;
- требуется ли повторная передача пакета на уровне алгоритма режима;
- можно ли оборвать вывод между пакетами без завершения текущей передачи;
- нужно ли обновлять MRAM после частичного вывода;
- нужно ли вести счётчик последнего выведенного пакета.
`
