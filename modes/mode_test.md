# Подалгоритм режима тестирования ППЗУ ПО НА NATALIA

## 1. Назначение документа

Настоящий документ описывает подалгоритм работы штатного ПО в режиме тестирования ППЗУ.

Документ определяет:
- условия входа в режим тестирования;
- последовательность подготовки и запуска теста банка NAND;
- правила записи, чтения и сравнения тестовых данных;
- формирование массива результатов теста;
- сохранение результатов теста в MRAM;
- условия штатного, командного и аварийного завершения режима.

Документ используется совместно с:
- основным описанием алгоритма;
- таблицей состояний и переходов;
- документом обработки команд CAN;
- документом аварийной и защитной логики;
- техническими требованиями на ПУ;
- описанием интерфейсов МК.

---

## 2. Назначение режима тестирования ППЗУ

Режим тестирования ППЗУ предназначен для проверки выбранного банка NAND.

Согласно протоколу CAN, в режиме тестирования ПО должно:
- включить питание выбранного банка NAND, если оно выключено;
- записать в банк фиксированную тестовую последовательность;
- выполнить чтение записанных данных;
- сравнить записанные и считанные данные;
- посчитать количество ошибок в каждом блоке банка;
- сформировать массив `Nerr(2048)` размером `2048 x 3 = 6144` байт;
- записать этот массив в MRAM;
- по завершении перейти в дежурный режим. :contentReference[oaicite:0]{index=0}

---

## 3. Условия входа в режим тестирования

### 3.1 Внешнее условие входа
Вход в режим выполняется по КУ:
- `CMD_TEST` (`MSG_ID = 0009h`)

### 3.2 Допустимый исходный режим
Вход допускается только из:
- `DUTY`

### 3.3 Обязательные условия входа
Для успешного запуска должны выполняться:
- команда корректна по формату и содержимому;
- выбран корректный идентификатор банка NAND;
- отсутствует ненулевой `MaskedAlarm`;
- выбранный банк доступен для питания и тестирования;
- служба NAND готова к выполнению полного цикла теста;
- служба MRAM готова сохранить результаты теста.

### 3.4 Причины отказа во входе
Запуск режима отклоняется, если:
- команда пришла не из `DUTY`;
- указан недопустимый идентификатор банка;
- содержимое команды некорректно;
- активна авария;
- не удалось подготовить NAND или MRAM;
- невозможно включить питание банка.

---

## 4. Входные параметры режима тестирования

Режим тестирования конфигурируется КУ `CMD_TEST`.

### 4.1 Аппаратная конфигурация
Команда задаёт:
- идентификатор используемого банка NAND:
  - `1` — `NAND1`
  - `2` — `NAND2`
- признак состояния питания выбранного банка после завершения теста:
  - `0` — выключить питание
  - `1` — оставить питание включённым

### 4.2 Что не задаётся командой
Команда не задаёт:
- параметры наблюдений;
- параметры ПЭД;
- параметры вывода данных;
- маску аварий;
- способ передачи результатов теста на борт.

---

## 5. Внутренние данные режима тестирования

Во время теста ПО должно поддерживать:

### 5.1 Постоянные на операцию
- выбранный банк NAND;
- требуемое состояние питания по завершении;
- размер тестируемого пространства;
- параметры тестовой последовательности.

### 5.2 Динамические данные
- текущее подсостояние режима;
- номер текущего тестируемого блока;
- буфер записываемых тестовых данных;
- буфер считанных данных;
- счётчик ошибок в текущем блоке;
- массив `Nerr[2048]`;
- флаг завершения записи;
- флаг завершения чтения;
- флаг завершения сравнения;
- флаг общей ошибки операции.

### 5.3 Служебные данные
- счётчик числа тестов выбранного банка;
- состояние питания банка;
- результат последнего теста для выбранного банка;
- указатель или область MRAM для хранения `Nerr`.

---

## 6. Общая последовательность входа в режим тестирования

При успешном принятии `CMD_TEST` ПО должно:

1. Выдать ТС «Квитанция» о принятии команды.
2. Сохранить параметры команды.
3. Выбрать целевой банк NAND.
4. Если питание банка выключено:
   - включить питание;
   - дождаться ожидаемого статуса питания, если требуется.
5. Подготовить сервис NAND к тесту.
6. Подготовить буферы тестирования.
7. Очистить массив `Nerr[2048]`.
8. Перевести верхний режим в `TEST`.
9. Сформировать ТС «Статус`.
10. Запустить процедуру тестирования.

---

## 7. Подсостояния внутри режима тестирования

| Подсостояние | Обозначение | Назначение |
|---|---|---|
| Подготовка входа | TEST_ENTER | Выбор банка, включение питания, подготовка буферов |
| Запись тестовых данных | TEST_WRITE | Запись тестового шаблона в NAND |
| Чтение тестовых данных | TEST_READ | Чтение данных из NAND |
| Сравнение и подсчёт ошибок | TEST_COMPARE | Сравнение и заполнение `Nerr` |
| Сохранение результатов | TEST_SAVE | Запись массива `Nerr` в MRAM |
| Штатное завершение | TEST_FINISH_OK | Обновление счётчиков и выход |
| Завершение по команде | TEST_FINISH_CMD | Выход по `CMD_DUTY` или `CMD_SHUTDOWN` |
| Завершение по аварии | TEST_FINISH_ALARM | Выход при `MaskedAlarm != 0` или ошибке |

---

## 8. Основные источники событий в режиме тестирования

### 8.1 Разрешённые КУ
В режиме тестирования разрешены:
- `CMD_SHUTDOWN`
- `CMD_DUTY`
- `CMD_STATUS_REQ`

Остальные КУ запрещены. :contentReference[oaicite:1]{index=1}

### 8.2 КТ
Все КТ в режиме тестирования игнорируются. :contentReference[oaicite:2]{index=2}

### 8.3 Внутренние события
Существенные события:
- подтверждение включения питания банка;
- завершение записи тестового шаблона;
- завершение чтения данных;
- завершение сравнения;
- завершение записи результатов в MRAM;
- ошибка QSPI;
- ошибка MRAM;
- ошибка питания NAND;
- появление `MaskedAlarm != 0`.

---

## 9. Общая логика теста

### 9.1 Смысл операции
Полный цикл теста включает:
1. запись фиксированной тестовой последовательности в выбранный банк;
2. чтение записанных данных;
3. сравнение данных;
4. подсчёт количества ошибок в каждом блоке по 256 кБ;
5. сохранение результатов.

### 9.2 Результат теста
Результатом является массив:
- `Nerr(2048)`
- по 3 байта на каждый блок
- общий размер 6144 байт

Этот массив далее может быть выдан на борт по КУ `CMD_TEST_RESULT`. :contentReference[oaicite:3]{index=3}

### 9.3 Приоритеты событий внутри режима
Рекомендуемый приоритет:
1. `MaskedAlarm != 0`
2. `CMD_SHUTDOWN`
3. внутренняя ошибка теста
4. завершение очередного этапа теста
5. `CMD_DUTY`
6. `CMD_STATUS_REQ`

---

## 10. Подготовка к тесту

### 10.1 Включение питания банка
Если целевой банк выключен, ПО должно:
1. установить соответствующий сигнал питания банка;
2. при необходимости дождаться подтверждения `PSON`;
3. при отсутствии ожидаемого подтверждения зафиксировать ошибку.

### 10.2 Инициализация буферов
До начала теста ПО должно:
- очистить `Nerr[2048]`;
- подготовить рабочие буферы записи и чтения;
- сбросить индекс текущего блока;
- сбросить признаки завершения фаз.

### 10.3 Подготовка NAND
ПО должно проверить готовность службы NAND:
- доступность банка;
- возможность записи;
- возможность чтения;
- отсутствие конфликта текущих операций.

---

## 11. Этап записи тестовых данных

### 11.1 Назначение
На этом этапе в тестируемый банк записывается фиксированная тестовая последовательность.

### 11.2 Действия алгоритма
ПО должно:
1. сформировать тестовый шаблон;
2. последовательно записывать его в блоки банка;
3. контролировать успешность записи;
4. после завершения записи перейти к этапу чтения.

### 11.3 Ошибки
На этом этапе ошибками считаются:
- отказ записи;
- сбой интерфейса QSPI;
- потеря питания банка;
- внутренняя ошибка службы NAND.

При таких ошибках режим должен завершаться аварийно.

---

## 12. Этап чтения тестовых данных

### 12.1 Назначение
На этом этапе ПО должно считывать ранее записанные тестовые данные из каждого блока банка.

### 12.2 Действия алгоритма
ПО должно:
1. последовательно читать данные из тестируемых блоков;
2. сохранять их во временный буфер;
3. передавать результат на этап сравнения.

### 12.3 Ошибки
Ошибками считаются:
- отказ чтения;
- ошибка интерфейса QSPI;
- ошибка доступа к банку;
- потеря питания банка.

---

## 13. Этап сравнения и подсчёта ошибок

### 13.1 Назначение
На этом этапе ПО должно сравнивать:
- ожидаемые тестовые данные;
- реально считанные данные.

### 13.2 Действия алгоритма
Для каждого блока ПО должно:
1. сравнить записанный и считанный массив;
2. подсчитать число несовпадений;
3. записать число ошибок в `Nerr[block_index]`;
4. перейти к следующему блоку.

### 13.3 Размерность результата
Для каждого блока хранится:
- 24-битное значение количества ошибок.

Это согласуется с форматом выдачи результатов теста ППЗУ, где на каждый блок отводится 3 байта. :contentReference[oaicite:4]{index=4}

---

## 14. Сохранение результатов теста

### 14.1 Назначение
После завершения тестирования массив `Nerr[2048]` должен быть записан в MRAM.

### 14.2 Действия алгоритма
ПО должно:
1. выбрать область хранения результатов для тестируемого банка;
2. записать туда массив `Nerr`;
3. подтвердить успешность записи;
4. подготовить результаты к последующей выдаче по `CMD_TEST_RESULT`.

### 14.3 Ошибки
Если запись в MRAM не удалась:
- это должно считаться ошибкой завершения теста;
- дальнейшее поведение нужно зафиксировать отдельно:
  - либо как переход в `ALARM`,
  - либо как внутреннюю ошибку без аварийного режима.

Это место пока лучше оставить как открытый вопрос.

---

## 15. Штатное завершение теста

### 15.1 Условие
Тест считается завершённым штатно, если:
- запись тестовых данных завершена;
- чтение завершено;
- сравнение выполнено для всех блоков;
- массив `Nerr` успешно сохранён.

### 15.2 Действия
При штатном завершении ПО должно:
1. обновить результаты последнего теста выбранного банка;
2. увеличить счётчик числа тестов выбранного банка;
3. при необходимости обновить MRAM-счётчики;
4. оставить питание банка включённым или отключить его по параметру команды;
5. перейти в `DUTY`;
6. сформировать ТС «Статус`.

Согласно CAN-протоколу, после завершения тестирования выполняется переход в дежурный режим. :contentReference[oaicite:5]{index=5}

---

## 16. Досрочное завершение по команде

### 16.1 Команда `CMD_DUTY`
При получении `CMD_DUTY` ПО должно:
1. выдать квитанцию;
2. выполнить завершение/останов теста по принятому правилу;
3. перейти в `DUTY`;
4. сформировать ТС «Статус`.

### 16.2 Команда `CMD_SHUTDOWN`
При получении `CMD_SHUTDOWN` ПО должно:
1. выдать квитанцию;
2. завершить или остановить тест по принятому правилу;
3. сохранить допустимый минимум служебных данных;
4. перейти в `SHUTDOWN`;
5. сформировать ТС «Статус`.

### 16.3 Открытый вопрос
Нужно отдельно решить, допустимо ли:
- прерывать тест в середине цикла;
- или требуется ждать завершения текущей фазы.

---

## 17. Завершение по аварии

### 17.1 Условие
Если в режиме тестирования `MaskedAlarm != 0`, ПО должно перейти в `ALARM`. :contentReference[oaicite:6]{index=6}

### 17.2 Действия
При аварийном завершении ПО должно:
1. прекратить процедуру тестирования;
2. отключить ПЭД;
3. отключить оба банка NAND;
4. зафиксировать переход в `ALARM`;
5. сформировать ТС «Статус`.

### 17.3 Типовые причины
Особенно важные причины в этом режиме:
- авария банка NAND;
- отказ питания банка;
- ошибка QSPI;
- выход температур, напряжений, токов из диапазонов. 

---

## 18. Обработка разрешённых команд в режиме тестирования

### 18.1 CMD_STATUS_REQ
При получении `CMD_STATUS_REQ`:
1. выдать ТС «Квитанция`;
2. выдать ТС «Статус`;
3. состояние не менять.

### 18.2 CMD_DUTY
При получении `CMD_DUTY`:
1. выдать ТС «Квитанция`;
2. инициировать завершение/останов теста;
3. перейти в `DUTY`;
4. выдать ТС «Статус`.

### 18.3 CMD_SHUTDOWN
При получении `CMD_SHUTDOWN`:
1. выдать ТС «Квитанция`;
2. инициировать завершение/останов теста;
3. перейти в `SHUTDOWN`;
4. выдать ТС «Статус`.

### 18.4 Остальные КУ
Остальные КУ:
- отклоняются;
- сопровождаются квитанцией с ошибкой;
- режим не меняют.

## 21. Открытые вопросы

В этом разделе фиксируются места, требующие уточнения:

- какой именно тестовый шаблон считается нормативным;
- можно ли прерывать тест между фазами или внутри фазы;
- считать ли ошибку сохранения `Nerr` в MRAM полноценной аварией;
- нужно ли частично сохранять результаты теста при досрочном завершении;
- нужно ли очищать тестируемый банк после завершения теста безусловно или это уже входит в сам тест;
- где именно в MRAM хранить результаты для двух банков.
