# Аварии и защитная логика ПО НА NATALIA

## 1. Назначение документа

Настоящий документ описывает:
- источники аварийных признаков в ПО НА NATALIA;
- правила формирования параметров `Аварийный статус НА` и `Замаскированный аварийный статус НА`;
- условия автоматического перехода в аварийный режим;
- ограничения на работу ПО в аварийном режиме;
- правила выхода из аварийного режима;
- защитные действия при отказах узлов и интерфейсов.

Документ используется совместно с:
- основным описанием алгоритма;
- таблицей состояний и переходов;
- документом обработки команд CAN;
- документами по отдельным режимам.

---

## 2. Общие положения

Аварийная логика ПО должна обеспечивать:
1. непрерывный контроль контролируемых параметров и состояний;
2. формирование аварийных признаков при обнаружении нештатных условий;
3. применение маски аварийного статуса;
4. автоматический перевод НА в аварийный режим при ненулевом замаскированном аварийном статусе;
5. защитное отключение ПЭД и банков NAND при входе в аварийный режим;
6. ограничение набора доступных команд в аварийном режиме;
7. возврат в дежурный режим только после выполнения условий выхода.

---

## 3. Основные определения

### 3.1 Аварийный статус НА
Параметр, содержащий битовые признаки аварийных состояний узлов и подсистем НА.

### 3.2 Маска аварийного статуса НА
Параметр конфигурации, задающий, какие аварийные признаки участвуют в принятии решения
о переходе в аварийный режим.

### 3.3 Замаскированный аварийный статус НА
Результат побитовой операции:

`MaskedAlarm = AlarmStatus AND AlarmMask`

Именно этот параметр участвует в логике автоматического перехода в аварийный режим.

### 3.4 Аварийный режим
Специальный режим работы ПО, в котором:
- оба банка ППЗУ отключены;
- ПЭД отключена;
- выполнение целевых режимов работы запрещено;
- разрешён только ограниченный набор команд.

---

## 4. Нормативные правила аварийной логики

### 4.1 Вход в аварийный режим
Если в режимах 1-5 `Замаскированный аварийный статус НА` становится отличным от 0,
ПО должно автоматически перейти в аварийный режим.

### 4.2 Состав действий при входе в аварийный режим
При входе в аварийный режим ПО должно:
1. прекратить выполнение текущего целевого режима;
2. отключить питание ПЭД;
3. отключить питание обоих банков NAND;
4. зафиксировать новый режим работы;
5. сформировать ТС «Статус».

### 4.3 Ограничения в аварийном режиме
В аварийном режиме запрещено включение режимов:
- стирания ППЗУ;
- тестирования ППЗУ;
- наблюдений;
- вывода данных.

### 4.4 Выход из аварийного режима
Выход из аварийного режима возможен только если:
- `Замаскированный аварийный статус НА == 0`.

### 4.5 Сброс аварийного статуса
Обнуление параметра `Аварийный статус НА` выполняется по КУ `Сброс аварийного статуса`.

### 4.6 Автоматическое обнуление
Аварийные признаки в `Аварийный статус НА` автоматически не сбрасываются.
Их обнуление требует отдельного действия алгоритма.

## 8. Правила вычисления маскированного аварийного статуса

### 8.1 Формула
Замаскированный аварийный статус вычисляется как:

`MaskedAlarm = AlarmStatus AND AlarmMask`

### 8.2 Момент пересчёта
Пересчёт должен выполняться:
- после изменения любого аварийного бита;
- после изменения маски аварий;
- после выполнения КУ `Сброс аварийного статуса`;
- после загрузки конфигурации из MRAM при старте ПО.

### 8.3 Реакция на результат пересчёта
- если `MaskedAlarm == 0`, аварийный режим не активируется или допускается выход из него;
- если `MaskedAlarm != 0`, в режимах 1-5 должен быть выполнен переход в `ALARM`.

---

## 9. Алгоритм обнаружения аварий

### 9.1 Общая схема
Аварийная логика должна включать:
1. сбор текущих измеренных и статусных параметров;
2. сравнение с порогами и ожидаемыми состояниями;
3. установку соответствующих битов `AlarmStatus`;
4. пересчёт `MaskedAlarm`;
5. при необходимости генерацию внутреннего события `E_MASKED_ALARM_SET`.

### 9.2 Источники запуска проверки
Проверка аварий может запускаться:
- периодически по событию `RTC 1 Гц`;
- по завершении длительной операции;
- по изменению сигналов статуса питания;
- по ошибке интерфейса ПЭД, NAND или USB;
- после применения новых настроек из КУ `Задание настроек`.

---

## 10. Действия при входе в аварийный режим

### 10.1 Общий алгоритм
При входе в `ALARM` ПО должно выполнить следующий порядок действий:

1. Зафиксировать причину входа в аварийный режим.
2. Остановить или прекратить текущую длительную операцию.
3. Запретить регистрацию событий в ПЭД.
4. Отключить питание ПЭД.
5. Отключить питание NAND1.
6. Отключить питание NAND2.
7. Обновить внутреннее состояние режима.
8. Сформировать ТС «Статус».
9. Продолжить работу в режиме `ALARM`.

### 10.2 Специальные замечания
- логика сохранения промежуточных данных при аварийном входе уточняется отдельно;
- приоритет аварийного перехода выше обычных команд и фоновых операций;
- при входе в `ALARM` возврат в предыдущий режим не выполняется автоматически.

---

## 11. Разрешённые команды в аварийном режиме

| Команда | Разрешена | Действие |
|---|---|---|
| Запрос телеметрии | Да | Выдать квитанцию и ТС «Телеметрия» |
| Запрос статуса | Да | Выдать квитанцию и ТС «Статус» |
| Задание настроек | Да | Обновить конфигурацию и маску |
| Сброс аварийного статуса | Да | Обнулить `AlarmStatus`, пересчитать `MaskedAlarm` |
| Выключение | Да | Перейти в `SHUTDOWN` |
| Остальные КУ | Нет | Квитанция с ошибкой |
| Все КТ | Нет | Игнорировать |

---

## 12. Правила выхода из аварийного режима

### 12.1 Необходимое условие
Выход из `ALARM` разрешён только при:

`MaskedAlarm == 0`

### 12.2 Способы достижения условия выхода
Это возможно в двух случаях:
1. изменена маска аварий так, что текущие аварийные признаки перестали участвовать в `MaskedAlarm`;
2. выполнен сброс аварийного статуса и после пересчёта `MaskedAlarm == 0`.

### 12.3 Действия при выходе
При выходе из `ALARM` ПО должно:
1. зафиксировать переход в дежурный режим;
2. не восстанавливать автоматически прежний целевой режим;
3. сформировать ТС «Статус`;
4. перейти в `DUTY`.

---

## 13. Аварийная логика по режимам

### 13.1 Дежурный режим
Возможны аварии:
- NAND;
- ПЭД;
- температуры;
- напряжения;
- токи.

При `MaskedAlarm != 0`:
- переход в `ALARM`.

### 13.2 Режим стирания ППЗУ
Дополнительно возможны:
- сбой QSPI;
- отказ питания банка NAND;
- ошибка процедуры стирания.

При `MaskedAlarm != 0`:
- прекращение операции;
- переход в `ALARM`.

### 13.3 Режим тестирования ППЗУ
Дополнительно возможны:
- ошибка записи/чтения NAND;
- ошибка тестовой последовательности;
- отказ питания банка NAND.

При `MaskedAlarm != 0`:
- прекращение операции;
- переход в `ALARM`.

### 13.4 Режим наблюдений
Дополнительно возможны:
- авария питания ПЭД;
- ошибка статуса ПЭД;
- ошибка обмена с ПЭД;
- отказ активного банка NAND.

При `MaskedAlarm != 0`:
- остановить наблюдение;
- отключить ПЭД и NAND;
- перейти в `ALARM`.

### 13.5 Режим вывода данных
Дополнительно возможны:
- отказ USB;
- отсутствие VBUS;
- тайм-аут передачи;
- отказ активного NAND.

При `MaskedAlarm != 0`:
- прекратить вывод;
- отключить ПЭД и NAND;
- перейти в `ALARM`.

---

## 14. Взаимодействие с КУ «Задание настроек»

Команда `Задание настроек` может изменять:
- пороги температур;
- пороги напряжений;
- пороги токов;
- маску аварийного статуса;
- другие конфигурационные параметры.

После успешного применения команды ПО должно:
1. обновить конфигурацию в ОЗУ;
2. записать конфигурацию в MRAM;
3. пересчитать `MaskedAlarm`;
4. при необходимости инициировать событие:
   - `E_MASKED_ALARM_SET`, если `MaskedAlarm` стал ненулевым;
   - `E_MASKED_ALARM_CLEAR`, если `MaskedAlarm` стал равным нулю в режиме `ALARM`.

---

## 15. Взаимодействие с КУ «Сброс аварийного статуса»

После получения КУ `Сброс аварийного статуса` в режиме `ALARM` ПО должно:
1. выдать квитанцию о принятии команды;
2. обнулить `AlarmStatus`;
3. пересчитать `MaskedAlarm`;
4. если `MaskedAlarm == 0`, инициировать переход в `DUTY`;
5. если `MaskedAlarm != 0`, остаться в `ALARM`.

Примечание:
если после сброса какие-либо контролируемые параметры всё ещё находятся вне допуска,
соответствующие биты могут быть выставлены повторно следующим циклом контроля.

```c

uint16_t alarm_get_mask(void);

void alarm_recalculate_masked(void);

void enter_alarm_mode(alarm_source_t source);
bool can_exit_alarm_mode(void);
```

- отключения питания;
- остановки текущих операций;
- отправки ТС «Статус»;
- логирования причины аварии.

- какие аварийные признаки могут сбрасываться автоматически, если вообще могут;
