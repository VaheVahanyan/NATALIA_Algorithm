# Подалгоритм режима тестирования NAND ПО НА NATALIA

## 1. Назначение документа

Настоящий документ описывает подалгоритм работы штатного ПО в режиме тестирования NAND.

Документ определяет:

- условия входа в режим тестирования;
- последовательность подготовки и запуска теста выбранного банка NAND;
- правила записи, чтения и сравнения тестовых данных;
- формирование массива результатов теста;
- сохранение результатов теста в MRAM;
- последующее стирание протестированного банка NAND;
- условия штатного, командного и аварийного завершения режима.

Документ используется совместно с:

- таблицей состояний и переходов;
- документом обработки команд CAN;
- документом допущений и открытых вопросов;
- документами по остальным режимам работы;
- диаграммами состояний Mermaid.

---

## 2. Назначение режима тестирования NAND

Режим `TEST` предназначен для проверки выбранного банка NAND.

В режиме `TEST` ПО должно:

- включить питание выбранного банка NAND, если оно выключено;
- записать в банк фиксированную тестовую последовательность;
- выполнить чтение записанных данных;
- сравнить записанные и считанные данные;
- подсчитать количество ошибок в каждом блоке банка;
- сформировать массив `Nerr(2048)`;
- записать этот массив в MRAM;
- выполнить последующее стирание протестированного банка;
- после завершения перейти в `DUTY`.

---

## 3. Условия входа в режим тестирования

### 3.1 Внешнее условие входа

Вход в режим выполняется по КУ:

- `CMD_TEST`

### 3.2 Допустимый исходный режим

Вход в режим допускается только из:

- `DUTY`

### 3.3 Обязательные условия входа

Для успешного запуска режима должны выполняться:

- команда корректна по формату и содержимому;
- выбран корректный идентификатор банка NAND;
- отсутствует ненулевой `MaskedAlarm`;
- выбранный банк доступен для питания и тестирования;
- служба NAND готова к выполнению полного цикла теста;
- служба MRAM доступна для сохранения результатов теста.

### 3.4 Причины отказа во входе

Запуск режима отклоняется, если:

- команда получена не из `DUTY`;
- указан недопустимый идентификатор банка NAND;
- содержимое команды некорректно;
- имеется активная авария;
- невозможно подготовить NAND или MRAM;
- невозможно включить питание банка.

---

## 4. Входные параметры режима тестирования

Режим тестирования конфигурируется КУ `CMD_TEST`.

### 4.1 Параметры команды

Команда задаёт:

- идентификатор используемого банка NAND:
    - `1` — `NAND1`
    - `2` — `NAND2`
- признак состояния питания выбранного банка после завершения теста:
    - `0` — отключить питание;
    - `1` — оставить питание включённым.

### 4.2 Ограничения команды

Команда `CMD_TEST` не задаёт:

- параметры наблюдений;
- параметры ПЭД;
- параметры вывода данных;
- маску аварий;
- формат выдачи результатов теста.

---

## 5. Внутренние данные режима тестирования

## 5.1 Постоянные на операцию

Во время режима `TEST` ПО должно поддерживать:

- выбранный банк NAND;
- требуемое состояние питания банка по завершении;
- размер тестируемого пространства;
- параметры тестовой последовательности.

## 5.2 Динамические данные

ПО должно поддерживать:

- текущее подсостояние режима;
- номер текущего тестируемого блока;
- буфер записываемых тестовых данных;
- буфер считанных данных;
- счётчик ошибок в текущем блоке;
- массив `Nerr[2048]`;
- флаг завершения записи;
- флаг завершения чтения;
- флаг завершения сравнения;
- флаг завершения записи результатов в MRAM;
- флаг завершения последующего стирания;
- флаг общей ошибки операции.

## 5.3 Служебные данные

ПО должно поддерживать:

- счётчик числа тестов выбранного банка;
- результат последнего валидного теста выбранного банка;
- область MRAM хранения результатов теста;
- состояние питания банка NAND.

---

## 6. Последовательность входа в режим тестирования

При успешном принятии `CMD_TEST` ПО должно:

1. Выдать ТС «Квитанция» о принятии команды.
2. Сохранить параметры команды.
3. Выбрать целевой банк NAND.
4. Если питание банка выключено, подать сигнал включения питания банка.
5. Дождаться подтверждения наличия питания банка по предусмотренному статусному сигналу.
6. Подготовить сервис NAND к тесту.
7. Подготовить буферы тестирования.
8. Очистить массив `Nerr[2048]`.
9. Перевести верхний режим в `TEST`.
10. Сформировать ТС «Статус».
11. Запустить процедуру тестирования.

<mark>На текущем этапе отсутствие ожидаемого подтверждения питания банка NAND в течение установленного тайм-аута
рассматривается как NAND-related отказ запуска режима.</mark>

---

## 7. Подсостояния режима тестирования

Для внутренней логики режим `TEST` разделяется на подэтапы:

| Подсостояние               | Обозначение         | Назначение                                            |
|----------------------------|---------------------|-------------------------------------------------------|
| Подготовка входа           | `TEST_ENTER`        | Выбор банка, включение питания, подготовка буферов    |
| Запись тестовых данных     | `TEST_WRITE`        | Запись тестовой последовательности в NAND             |
| Чтение тестовых данных     | `TEST_READ`         | Чтение ранее записанных данных                        |
| Сравнение и подсчёт ошибок | `TEST_COMPARE`      | Сравнение записанного и считанного, заполнение `Nerr` |
| Сохранение результатов     | `TEST_SAVE`         | Запись массива `Nerr` в MRAM                          |
| Последующее стирание       | `TEST_ERASE`        | Стирание протестированного банка NAND                 |
| Штатное завершение         | `TEST_FINISH_OK`    | Обновление служебных данных и выход                   |
| Завершение по команде      | `TEST_FINISH_CMD`   | Выход по `CMD_DUTY` или `CMD_SHUTDOWN`                |
| Завершение по аварии       | `TEST_FINISH_ALARM` | Выход при `MaskedAlarm != 0` или внутренней ошибке    |

---

## 8. Основные источники событий в режиме тестирования

## 8.1 Разрешённые КУ

В режиме `TEST` разрешены:

- `CMD_SHUTDOWN`
- `CMD_DUTY`
- `CMD_STATUS_REQ`

Остальные КУ запрещены и сопровождаются квитанцией с ошибкой.

## 8.2 КТ

Все КТ в режиме `TEST` игнорируются.

## 8.3 Внутренние события

В режиме `TEST` существенны:

- подтверждение включения питания банка;
- завершение записи тестовой последовательности;
- завершение чтения данных;
- завершение сравнения данных;
- завершение записи результатов в MRAM;
- завершение последующего стирания банка;
- ошибка `QSPI`;
- ошибка MRAM;
- ошибка питания NAND;
- появление `MaskedAlarm != 0`.

---

## 9. Основной цикл режима тестирования

### 9.1 Общая логика

Режим `TEST` является длительной многофазной операцией.  
После запуска режима ПО должно последовательно выполнить:

1. запись тестовой последовательности;
2. чтение данных;
3. сравнение данных и заполнение массива `Nerr`;
4. запись массива `Nerr` в MRAM;
5. стирание протестированного банка;
6. завершение режима.

### 9.2 Приоритет обработки событий

В режиме `TEST` использовать следующий логический приоритет:

1. `MaskedAlarm != 0`
2. `CMD_SHUTDOWN`
3. внутренняя ошибка теста
4. завершение текущей фазы
5. `CMD_DUTY`
6. `CMD_STATUS_REQ`

---

## 10. Подготовка к тестированию

### 10.1 Включение питания банка

Если целевой банк NAND выключен, ПО должно:

1. установить сигнал включения питания банка;
2. дождаться ожидаемого состояния сигнала подтверждения питания;
3. при отсутствии подтверждения в течение установленного тайм-аута зафиксировать отказ включения банка.

### 10.2 Инициализация буферов

До начала теста ПО должно:

- очистить массив `Nerr[2048]`;
- подготовить рабочие буферы записи и чтения;
- сбросить индекс текущего блока;
- сбросить признаки завершения фаз.

### 10.3 Подготовка NAND

ПО должно проверить:

- доступность банка NAND;
- возможность записи;
- возможность чтения;
- отсутствие конфликта с текущими операциями.

---

## 11. Этап записи тестовых данных

### 11.1 Назначение

На этапе `TEST_WRITE` в тестируемый банк записывается фиксированная тестовая последовательность.

### 11.2 Действия алгоритма

ПО должно:

1. сформировать тестовый шаблон;
2. последовательно записывать его в блоки банка NAND;
3. контролировать успешность записи;
4. после завершения записи перейти к этапу чтения.

### 11.3 Ошибки этапа записи

Ошибками этапа считаются:

- отказ записи;
- сбой `QSPI`;
- потеря питания банка NAND;
- внутренняя ошибка службы NAND.

---

## 12. Этап чтения тестовых данных

### 12.1 Назначение

На этапе `TEST_READ` ПО должно считывать ранее записанные тестовые данные из каждого блока банка NAND.

### 12.2 Действия алгоритма

ПО должно:

1. последовательно читать данные из тестируемых блоков;
2. сохранять их во временный буфер;
3. передавать считанные данные на этап сравнения.

### 12.3 Ошибки этапа чтения

Ошибками этапа считаются:

- отказ чтения;
- ошибка `QSPI`;
- ошибка доступа к банку;
- потеря питания банка NAND.

---

## 13. Этап сравнения и подсчёта ошибок

### 13.1 Назначение

На этапе `TEST_COMPARE` ПО должно сравнивать:

- ожидаемые тестовые данные;
- фактически считанные данные.

### 13.2 Действия алгоритма

Для каждого блока ПО должно:

1. сравнить записанный и считанный массив;
2. подсчитать число несовпадений;
3. записать число ошибок в `Nerr[block_index]`;
4. перейти к следующему блоку.

### 13.3 Представление результата

Для каждого блока в `Nerr` хранится 24-битное значение числа ошибок.

---

## 14. Сохранение результатов теста

### 14.1 Назначение

После завершения сравнения массив `Nerr[2048]` должен быть записан в MRAM.

### 14.2 Действия алгоритма

ПО должно:

1. выбрать область хранения результата для тестируемого банка;
2. записать массив `Nerr` в MRAM;
3. подтвердить успешность записи;
4. подготовить результат к последующей выдаче по `CMD_TEST_RESULT`.

### 14.3 Правило валидности результата

Только полностью сохранённый массив `Nerr` считается валидным результатом теста.

### 14.4 Ошибка записи в MRAM

<mark>На текущем этапе при ошибке записи массива `Nerr` в MRAM результат теста считается недействительным. Окончательная
классификация такой ошибки как внутренней ошибки режима либо как отдельной аварийной ситуации подлежит дополнительному
согласованию.</mark>

---

## 15. Последующее стирание протестированного банка

### 15.1 Общее правило

После успешного завершения этапов записи, чтения, сравнения и сохранения массива `Nerr` ПО должно выполнить стирание
протестированного банка NAND.

### 15.2 Действия алгоритма

На этапе `TEST_ERASE` ПО должно:

1. инициировать стирание протестированного банка;
2. дождаться завершения стирания;
3. подтвердить успешность операции;
4. только после этого перейти к штатному завершению режима.

### 15.3 Ошибка этапа стирания

Если последующее стирание не завершилось успешно, режим `TEST` не считается завершённым штатно.

<mark>На текущем этапе полный цикл режима `TEST` считается завершённым штатно только после успешного стирания
протестированного банка NAND.</mark>

---

## 16. Штатное завершение режима тестирования

### 16.1 Условие завершения

Режим `TEST` считается завершённым штатно, если:

- запись тестовых данных завершена;
- чтение данных завершено;
- сравнение выполнено для всех блоков;
- массив `Nerr` успешно сохранён в MRAM;
- последующее стирание банка NAND завершено успешно.

### 16.2 Действия при штатном завершении

При штатном завершении ПО должно:

1. обновить результаты последнего теста выбранного банка;
2. увеличить счётчик числа тестов выбранного банка;
3. при необходимости обновить служебные данные в MRAM;
4. оставить питание банка включённым или отключить его по параметру команды;
5. перейти в `DUTY`;
6. сформировать ТС «Статус».

---

## 17. Досрочное завершение по команде

## 17.1 Завершение по `CMD_DUTY`

При получении `CMD_DUTY` ПО должно:

1. выдать ТС «Квитанция»;
2. инициировать завершение режима;
3. после завершения перейти в `DUTY`;
4. сформировать ТС «Статус».

## 17.2 Завершение по `CMD_SHUTDOWN`

При получении `CMD_SHUTDOWN` ПО должно:

1. выдать ТС «Квитанция»;
2. инициировать завершение режима;
3. после завершения перейти в `SHUTDOWN`;
4. сформировать ТС «Статус».

## 17.3 Правило досрочного завершения

<mark>На текущем этапе принимается, что режим `TEST` не прерывается посреди элементарной низкоуровневой операции внутри
текущей фазы. Завершение по `CMD_DUTY` или `CMD_SHUTDOWN` выполняется после завершения текущей фазы теста.</mark>

### 17.4 Частичный результат теста

Частично выполненный тест не считается валидным результатом и не должен выдаваться по `CMD_TEST_RESULT`.

---

## 18. Аварийное завершение режима тестирования

### 18.1 Условие

Если в режиме `TEST` выполняется условие `MaskedAlarm != 0`, ПО должно выполнить переход в `ALARM`.

### 18.2 Действия при аварийном завершении

При аварийном завершении режима `TEST` ПО должно:

1. прекратить выполнение тестовой последовательности;
2. отключить ПЭД;
3. отключить питание `NAND1`;
4. отключить питание `NAND2`;
5. зафиксировать переход в `ALARM`;
6. сформировать ТС «Статус».

### 18.3 Типовые причины аварийного завершения

Для режима `TEST` особенно существенны:

- отказ питания банка NAND;
- ошибка `QSPI`;
- отказ записи;
- отказ чтения;
- ошибка последующего стирания;
- выход контролируемых параметров за допустимые пределы.

---

## 19. Обработка разрешённых команд в режиме тестирования

## 19.1 `CMD_STATUS_REQ`

При получении `CMD_STATUS_REQ` ПО должно:

1. выдать ТС «Квитанция»;
2. сформировать ТС «Статус»;
3. не изменять режим.

## 19.2 `CMD_DUTY`

При получении `CMD_DUTY` ПО должно:

1. выдать ТС «Квитанция»;
2. инициировать завершение режима;
3. выполнить выход в `DUTY` по принятому правилу завершения;
4. сформировать ТС «Статус».

## 19.3 `CMD_SHUTDOWN`

При получении `CMD_SHUTDOWN` ПО должно:

1. выдать ТС «Квитанция»;
2. инициировать завершение режима;
3. выполнить выход в `SHUTDOWN` по принятому правилу завершения;
4. сформировать ТС «Статус».

## 19.4 Остальные КУ

Остальные КУ:

- отклоняются;
- сопровождаются ТС «Квитанция» с ошибкой;
- не изменяют состояние режима.

---

## 20. Контроль аппаратного состояния в режиме тестирования

### 20.1 Контроль статусов питания

В режиме `TEST` ПО должно контролировать:

- сигнал включения питания активного банка;
- сигнал подтверждения питания активного банка;
- наличие рассогласования между требуемым и фактическим состоянием питания.

### 20.2 Потеря подтверждения питания в процессе теста

Если в процессе тестирования пропадает подтверждение питания активного банка, это должно трактоваться как аварийное
событие, приводящее к аварийному завершению режима.

---
