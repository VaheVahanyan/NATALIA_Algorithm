# Подалгоритм дежурного режима ПО НА NATALIA

## 1. Назначение документа

Настоящий документ описывает подалгоритм работы штатного ПО в дежурном режиме.

Документ определяет:

- назначение дежурного режима;
- условия входа в режим;
- состав аппаратной конфигурации режима;
- допустимые команды;
- действия ПО в режиме ожидания;
- переходы из дежурного режима в целевые режимы;
- обработку аварийных условий и команды выключения.

Документ используется совместно с:

- таблицей состояний и переходов;
- документом обработки команд CAN;
- документами по остальным режимам работы;
- документом допущений и открытых вопросов;
- диаграммами состояний Mermaid.

---

## 2. Назначение дежурного режима

Дежурный режим является базовым режимом работы НА.

В режиме `DUTY` ПО должно:

- ожидать входящие КУ;
- отклонять недопустимые команды;
- игнорировать все КТ;
- выдавать ТС «Статус» и ТС «Телеметрия» по разрешенным КУ;
- обновлять RTC по КУ `CMD_SET_TIME`;
- принимать новые настройки по КУ `CMD_SET_CFG`;
- запускать целевые режимы;
- контролировать аварийные признаки;
- контролировать аппаратное состояние узлов, используемых в дежурной конфигурации;
- по КУ `CMD_SHUTDOWN` переходить в `SHUTDOWN`.

Дежурный режим является центральной точкой маршрутизации верхнеуровневого автомата состояний.

---

## 3. Условия входа в дежурный режим

### 3.1 Источники входа

Переход в `DUTY` выполняется:

- после завершения инициализации при `MaskedAlarm == 0`;
- после штатного завершения `ERASE`;
- после штатного завершения `TEST`;
- после завершения `OBSERVE` по команде или по переполнению NAND;
- после штатного завершения `DUMP`;
- после выхода из `ALARM`, если `MaskedAlarm == 0`;
- после завершения `SHUTDOWN` не выполняется;
- по КУ `CMD_DUTY` из `ERASE`, `TEST`, `OBSERVE`, `DUMP`;
- по КУ `CMD_DUTY` из самого `DUTY`.

### 3.2 Условия успешного входа

Для входа в `DUTY` должны выполняться:

- отсутствует условие продолжения работы в `ALARM`;
- завершены или остановлены операции предыдущего режима;
- аппаратная конфигурация приведена к конфигурации дежурного режима;
- запись НИ остановлена;
- вывод данных остановлен;
- ПЭД и NAND переведены в состояние, соответствующее логике входа в `DUTY`.

### 3.3 Действия при входе

При входе в `DUTY` ПО должно:

1. зафиксировать предыдущий режим;
2. обновить текущий верхнеуровневый режим;
3. применить аппаратную конфигурацию дежурного режима;
4. пересчитать актуальные служебные параметры режима;
5. сформировать ТС «Статус».

---

## 4. Аппаратная конфигурация дежурного режима

### 4.1 Общие положения

Дежурный режим не означает обязательного отключения всех управляемых узлов.

В режиме `DUTY` допускается аппаратная конфигурация, в которой:

- один из банков NAND включен;
- ПЭД включена;
- ПЭД переведена в режим пониженного энергопотребления.

При этом в режиме `DUTY` не выполняется целевая научная активность.

### 4.2 Состав используемых интерфейсов

В режиме `DUTY` используются:

- `CAN1` — для приема КУ, приема КТ и передачи ТС;
- `SPI1`, `SPI3` — для работы с MRAM;
- `I2C1`, `I2C2` — для контроля параметров и датчиков;
- `ADC` — для чтения аналогового термодатчика;
- `PED_REG` и линии управления ПЭД — только если текущая дежурная конфигурация требует включенной ПЭД;
- интерфейс выбранного банка NAND — только если текущая дежурная конфигурация требует включенного банка NAND.

Интерфейсы `CAN2`, `I2C3`, `USART2` в штатной логике режима `DUTY` не используются.

### 4.3 Требования к конфигурации

ПО должно уметь:

- включать или выключать выбранный банк NAND;
- подтверждать состояние питания выбранного банка NAND по `PU_NANDx_PSON`;
- включать или выключать ПЭД;
- подтверждать состояние питания ПЭД по `PED_PSON`;
- включать или выключать `PED_SLEEP`, если ПЭД включена;
- управлять `PED_INHIBIT` в соответствии с дежурной конфигурацией;
- не запускать наблюдения, тестирование, стирание и вывод данных без соответствующей КУ.

### 4.4 Ограничения режима

В режиме `DUTY` не должны выполняться:

- регистрация событий ПЭД как научных событий;
- запись научной информации в NAND;
- вывод накопленных пакетов НИ;
- тестирование NAND;
- стирание NAND;
- запись КТ в поток НИ.

---

## 5. Допустимые команды в дежурном режиме

В режиме `DUTY` разрешены:

- `CMD_TELEM_REQ`;
- `CMD_STATUS_REQ`;
- `CMD_SET_TIME`;
- `CMD_OBSERVE_START`;
- `CMD_DUTY`;
- `CMD_DUMP`;
- `CMD_SET_CFG`;
- `CMD_ERASE`;
- `CMD_TEST`;
- `CMD_TEST_RESULT`;
- `CMD_SHUTDOWN`.

В режиме `DUTY` запрещены:

- `CMD_OBSERVE_CTRL`;
- `CMD_RESET_ALARM`.

Все КТ в режиме `DUTY` игнорируются.

---

## 6. Основные источники событий в дежурном режиме

### 6.1 Внешние команды управления

Основной набор событий в `DUTY` составляют разрешенные КУ.

### 6.2 Команды телеметрии

Все КТ в `DUTY` игнорируются без выдачи квитанции.

### 6.3 Внутренние события

В режиме `DUTY` существенны следующие внутренние события:

- изменение `MaskedAlarm`;
- изменение статусных сигналов питания;
- обновление контролируемых параметров;
- завершение аппаратной перестройки при входе в дежурную конфигурацию, если перестройка многошаговая.

---

## 7. Основной цикл дежурного режима

### 7.1 Общая последовательность

В дежурном режиме ПО должно:

1. находиться в состоянии ожидания команд;
2. проверять допустимость входящих КУ;
3. выполнять разрешенные сервисные действия;
4. запускать целевые режимы по соответствующим КУ;
5. игнорировать все КТ;
6. контролировать `MaskedAlarm`;
7. контролировать аппаратное состояние узлов, задействованных в дежурной конфигурации.

### 7.2 Приоритет обработки событий

В режиме `DUTY` используется следующий логический приоритет:

1. `MaskedAlarm != 0`;
2. `CMD_SHUTDOWN`;
3. команды запуска целевых режимов;
4. `CMD_SET_CFG`;
5. `CMD_SET_TIME`;
6. `CMD_STATUS_REQ`;
7. `CMD_TELEM_REQ`;
8. остальные допустимые сервисные команды.

---

## 8. Обработка разрешенных команд в дежурном режиме

### 8.1 `CMD_STATUS_REQ`

При получении `CMD_STATUS_REQ` ПО должно:

1. выдать ТС «Квитанция»;
2. выдать ТС «Статус»;
3. остаться в `DUTY`.

---

### 8.2 `CMD_TELEM_REQ`

При получении `CMD_TELEM_REQ` ПО должно:

1. выдать ТС «Квитанция»;
2. выдать ТС «Телеметрия»;
3. остаться в `DUTY`.

---

### 8.3 `CMD_SET_TIME`

При получении `CMD_SET_TIME` ПО должно:

1. выдать ТС «Квитанция»;
2. обновить приборное время RTC;
3. обновить связанные программные структуры времени;
4. остаться в `DUTY`.

---

### 8.4 `CMD_SET_CFG`

При получении `CMD_SET_CFG` ПО должно:

1. выдать ТС «Квитанция»;
2. проверить допустимость параметров новой конфигурации;
3. записать новую конфигурацию в первую копию MRAM;
4. выдержать интервал завершения записи;
5. записать новую конфигурацию во вторую копию MRAM;
6. сделать новую конфигурацию текущей рабочей конфигурацией;
7. пересчитать `MaskedAlarm`.

Если после применения конфигурации `MaskedAlarm == 0`, ПО остается в `DUTY`.

Если после применения конфигурации `MaskedAlarm != 0`, ПО должно:

1. сформировать ТС «Статус»;
2. перейти в `ALARM`.

---

### 8.5 `CMD_OBSERVE_START`

При получении `CMD_OBSERVE_START` ПО должно:

1. выдать ТС «Квитанция`, если команда допустима;
2. проверить конфигурацию запуска режима наблюдений;
3. проверить допустимость выбранного банка NAND;
4. при успехе передать управление подалгоритму `OBSERVE`;
5. после перехода в `OBSERVE` сформировать ТС «Статус`.

Если выбранный банк NAND уже отмечен как переполненный либо недоступный для запуска наблюдений, команда отклоняется.

---

### 8.6 `CMD_ERASE`

При получении `CMD_ERASE` ПО должно:

1. выдать ТС «Квитанция»;
2. проверить параметры стирания;
3. при успехе передать управление подалгоритму `ERASE`;
4. после перехода в `ERASE` сформировать ТС «Статус».

---

### 8.7 `CMD_TEST`

При получении `CMD_TEST` ПО должно:

1. выдать ТС «Квитанция»;
2. проверить параметры тестирования;
3. при успехе передать управление подалгоритму `TEST`;
4. после перехода в `TEST` сформировать ТС «Статус`.

---

### 8.8 `CMD_DUMP`

При получении `CMD_DUMP` ПО должно:

1. выдать ТС «Квитанция»;
2. проверить параметры вывода;
3. при успехе передать управление подалгоритму `DUMP`;
4. после перехода в `DUMP` сформировать ТС «Статус`.

---

### 8.9 `CMD_TEST_RESULT`

При получении `CMD_TEST_RESULT` ПО должно:

1. выдать ТС «Квитанция»;
2. считать сохраненный валидный результат теста для выбранного банка;
3. выдать ТС «Результаты теста ППЗУ»;
4. остаться в `DUTY`.

---

### 8.10 `CMD_DUTY`

При получении `CMD_DUTY` в самом `DUTY` ПО должно:

1. выдать ТС «Квитанция»;
2. не изменять текущий режим;
3. не выполнять дополнительную реконфигурацию режима.

---

### 8.11 `CMD_SHUTDOWN`

При получении `CMD_SHUTDOWN` ПО должно:

1. выдать ТС «Квитанция»;
2. передать управление подалгоритму `SHUTDOWN`;
3. после перехода в `SHUTDOWN` сформировать ТС «Статус`.

---

## 9. Обработка запрещенных команд

### 9.1 `CMD_OBSERVE_CTRL`

Команда `CMD_OBSERVE_CTRL` запрещена в `DUTY` и должна:

- быть отклонена;
- сопровождаться ТС «Квитанция» с ошибкой `ERR_MODE`.

### 9.2 `CMD_RESET_ALARM`

Команда `CMD_RESET_ALARM` запрещена в `DUTY` и должна:

- быть отклонена;
- сопровождаться ТС «Квитанция» с ошибкой `ERR_MODE`.

### 9.3 КТ

Все КТ в `DUTY`:

- игнорируются;
- не сопровождаются квитанцией;
- не изменяют текущий режим.

---

## 10. Переходы из дежурного режима

### 10.1 Переход в `OBSERVE`

Переход выполняется по:

- `CMD_OBSERVE_START`.

### 10.2 Переход в `ERASE`

Переход выполняется по:

- `CMD_ERASE`.

### 10.3 Переход в `TEST`

Переход выполняется по:

- `CMD_TEST`.

### 10.4 Переход в `DUMP`

Переход выполняется по:

- `CMD_DUMP`.

### 10.5 Переход в `ALARM`

Переход выполняется автоматически при:

- `MaskedAlarm != 0`.

### 10.6 Переход в `SHUTDOWN`

Переход выполняется по:

- `CMD_SHUTDOWN`.

---

## 11. Переход в аварийный режим из DUTY

### 11.1 Условие

Если в режиме `DUTY` параметр `MaskedAlarm` стал ненулевым, ПО должно выполнить переход в `ALARM`.

### 11.2 Действия

При переходе в `ALARM` из `DUTY` ПО должно:

1. зафиксировать причину перехода;
2. привести систему к безопасной конфигурации;
3. сформировать ТС «Статус»;
4. передать управление подалгоритму `ALARM`.

---

## 12. Контроль аппаратного состояния в режиме DUTY

### 12.1 Общие правила

В режиме `DUTY` ПО должно контролировать состояние управляемых узлов и статусных сигналов, относящихся к:

- ПЭД, если она включена в дежурной конфигурации;
- выбранному банку NAND, если он включен в дежурной конфигурации;
- линиям подтверждения питания используемых узлов;
- контролируемым температурным и электрическим параметрам.

### 12.2 Подтверждение изменения состояния

Если для соответствующего узла предусмотрен сигнал подтверждения статуса питания, ПО должно использовать его для
контроля корректности включения или отключения узла.

### 12.3 Отсутствие подтверждения

Если требуемое состояние включенного либо выключаемого узла не подтверждается в течение установленного тайм-аута, ПО
должно сформировать соответствующий аварийный признак и пересчитать `MaskedAlarm`.

### 12.4 Дежурная конфигурация с включенной ПЭД

Если ПЭД включена в дежурной конфигурации, ПО должно:

- контролировать `PED_PSON`;
- поддерживать требуемое состояние `PED_SLEEP`;
- поддерживать требуемое состояние `PED_INHIBIT`;
- не использовать триггеры ПЭД как источник формирования НИ.

### 12.5 Дежурная конфигурация с включенным банком NAND

Если один банк NAND включен в дежурной конфигурации, ПО должно:

- контролировать `PU_NANDx_PSON`;
- не выполнять запись НИ;
- не выполнять тестирование, стирание и вывод данных без отдельной КУ.
