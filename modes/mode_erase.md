# Подалгоритм режима стирания NAND ПО НА NATALIA

## 1. Назначение документа

Настоящий документ описывает подалгоритм работы штатного ПО в режиме стирания NAND.

Документ определяет:

- условия входа в режим стирания;
- последовательность действий при запуске стирания;
- правила управления питанием выбранного банка NAND;
- внутренние события режима;
- условия штатного завершения;
- условия досрочного завершения по команде;
- условия аварийного завершения;
- действия при выходе из режима.

Документ используется совместно с:

- таблицей состояний и переходов;
- документом обработки команд CAN;
- документом допущений и открытых вопросов;
- документами по остальным режимам работы;
- диаграммами состояний Mermaid.

---

## 2. Назначение режима стирания NAND

Режим `ERASE` предназначен для:

- включения питания выбранного банка NAND, если оно ещё не включено;
- выполнения процедуры стирания выбранного банка NAND;
- обновления служебной информации о состоянии банка;
- опционального отключения питания банка по завершении;
- последующего возврата в `DUTY`.

Режим стирания запускается только из `DUTY`.

После штатного завершения стирания выполняется автоматический переход в `DUTY`.

---

## 3. Условия входа в режим стирания

### 3.1 Внешнее условие входа

Вход в режим выполняется по КУ:

- `CMD_ERASE`

### 3.2 Допустимый исходный режим

Вход в режим допускается только из:

- `DUTY`

### 3.3 Обязательные условия входа

Для успешного запуска режима должны выполняться:

- команда корректна по длине и содержимому;
- выбран корректный идентификатор банка NAND;
- отсутствует ненулевой `MaskedAlarm`;
- выбранный банк доступен для управления;
- служба NAND готова к запуску операции стирания.

### 3.4 Причины отказа во входе

Запуск режима должен быть отклонён, если:

- команда получена не из `DUTY`;
- указан недопустимый идентификатор банка;
- содержимое команды некорректно;
- уже имеется активная авария;
- невозможно включить питание банка;
- невозможно начать процедуру стирания.

---

## 4. Входные параметры режима стирания

Режим стирания конфигурируется КУ `CMD_ERASE`.

### 4.1 Параметры команды

Команда задаёт:

- идентификатор используемого банка NAND:
    - `1` — `NAND1`
    - `2` — `NAND2`
- признак состояния питания выбранного банка по завершении стирания:
    - `0` — отключить питание;
    - `1` — оставить питание включённым.

### 4.2 Ограничения команды

Команда `CMD_ERASE` не управляет:

- ПЭД;
- вторым банком NAND;
- параметрами наблюдений;
- параметрами вывода данных;
- маской аварий.

---

## 5. Внутренние данные режима стирания

## 5.1 Постоянные на операцию

Во время режима `ERASE` ПО должно поддерживать:

- выбранный банк NAND;
- требуемое состояние питания банка после завершения;
- источник запуска операции;
- параметры текущей операции стирания.

## 5.2 Динамические данные

ПО должно поддерживать:

- текущее подсостояние процедуры;
- флаг включённого питания банка;
- флаг успешного старта стирания;
- флаг завершения стирания;
- флаг ошибки стирания;
- код внутренней причины отказа.

## 5.3 Служебные данные

ПО должно поддерживать:

- счётчик числа стираний выбранного банка;
- состояние сигнала управления питанием выбранного банка;
- состояние сигнала подтверждения питания выбранного банка.

---

## 6. Последовательность входа в режим стирания

При успешном принятии `CMD_ERASE` ПО должно:

1. Выдать ТС «Квитанция» о принятии команды.
2. Сохранить параметры команды.
3. Выбрать целевой банк NAND.
4. Если питание банка выключено, подать сигнал включения питания банка.
5. Дождаться подтверждения наличия питания банка по предусмотренному статусному сигналу.
6. Подготовить сервис NAND к операции стирания.
7. Перевести верхний режим в `ERASE`.
8. Сформировать ТС «Статус».
9. Запустить процедуру стирания банка.

<mark>На текущем этапе отсутствие ожидаемого подтверждения питания банка NAND в течение установленного тайм-аута
рассматривается как аварийное условие и приводит к NAND-related отказу запуска режима.</mark>

---

## 7. Подсостояния режима стирания

Для внутренней логики режим `ERASE` разделяется на подэтапы:

| Подсостояние          | Обозначение          | Назначение                                          |
|-----------------------|----------------------|-----------------------------------------------------|
| Подготовка входа      | `ERASE_ENTER`        | Проверка параметров, выбор банка, включение питания |
| Запуск стирания       | `ERASE_START`        | Запуск низкоуровневой операции стирания             |
| Ожидание завершения   | `ERASE_WAIT`         | Ожидание завершения процедуры стирания              |
| Штатное завершение    | `ERASE_FINISH_OK`    | Обновление служебных данных и выход                 |
| Завершение по команде | `ERASE_FINISH_CMD`   | Досрочный выход по `CMD_DUTY` или `CMD_SHUTDOWN`    |
| Завершение по аварии  | `ERASE_FINISH_ALARM` | Немедленное завершение при `MaskedAlarm != 0`       |

---

## 8. Основные источники событий в режиме стирания

## 8.1 Разрешённые КУ

В режиме `ERASE` разрешены:

- `CMD_SHUTDOWN`
- `CMD_DUTY`
- `CMD_STATUS_REQ`

Остальные КУ запрещены и сопровождаются квитанцией с ошибкой.

## 8.2 КТ

Все КТ в режиме `ERASE` игнорируются.

## 8.3 Внутренние события

В режиме `ERASE` существенны:

- подтверждение включения питания банка;
- успешный старт стирания;
- завершение стирания;
- ошибка стирания;
- ошибка интерфейса `QSPI`;
- изменение статуса питания банка;
- появление `MaskedAlarm != 0`.

---

## 9. Основной цикл режима стирания

### 9.1 Общая логика

Режим `ERASE` является длительной операцией.  
После запуска стирания ПО должно:

- ожидать завершения операции;
- контролировать состояние банка и линии питания;
- обслуживать разрешённые команды;
- контролировать аварийные условия;
- по завершении перейти в `DUTY` либо в иной режим в соответствии с событием завершения.

### 9.2 Приоритет обработки событий

В режиме `ERASE` использовать следующий логический приоритет:

1. `MaskedAlarm != 0`
2. `CMD_SHUTDOWN`
3. ошибка стирания или ошибка `QSPI`
4. `E_ERASE_DONE`
5. `CMD_DUTY`
6. `CMD_STATUS_REQ`

---

## 10. Подготовка и запуск стирания

### 10.1 Включение питания банка

Если целевой банк NAND выключен, ПО должно:

1. установить сигнал включения питания банка;
2. дождаться ожидаемого состояния сигнала подтверждения питания;
3. при отсутствии подтверждения в течение установленного тайм-аута зафиксировать отказ включения банка.

### 10.2 Запуск операции стирания

После подготовки питания ПО должно:

1. инициализировать доступ к банку NAND;
2. запустить процедуру стирания;
3. перевести режим в фазу ожидания завершения.

### 10.3 Ошибка запуска стирания

Если процедура стирания не может быть запущена из-за недоступности банка NAND, отказа питания или ошибки `QSPI`, запуск
режима считается неуспешным.

<mark>На текущем этапе ошибка запуска стирания рассматривается как NAND-related отказ и связывается с аварийной логикой
NAND.</mark>

---

## 11. Ожидание завершения стирания

После запуска стирания ПО должно находиться в фазе ожидания до наступления одного из событий:

- операция завершилась успешно;
- возникла ошибка операции;
- поступила команда `CMD_DUTY`;
- поступила команда `CMD_SHUTDOWN`;
- возникло условие `MaskedAlarm != 0`.

В этой фазе ПО может:

- отвечать на `CMD_STATUS_REQ`;
- контролировать состояние банка;
- контролировать аварийные признаки;
- не запускать новые целевые режимы.

---

## 12. Штатное завершение режима стирания

### 12.1 Условие завершения

Операция считается завершённой штатно при получении внутреннего события завершения стирания.

### 12.2 Действия при штатном завершении

При штатном завершении ПО должно:

1. отметить, что стирание выбранного банка завершено;
2. обновить информацию о состоянии банка;
3. сбросить признак переполнения банка, если такая логика используется;
4. увеличить счётчик числа стираний соответствующего банка;
5. при необходимости обновить служебные данные в MRAM;
6. оставить питание банка включённым или отключить его в соответствии с командой;
7. перейти в `DUTY`;
8. сформировать ТС «Статус».

### 12.3 Сброс признака переполнения банка

<mark>На текущем этапе принимается, что после успешного стирания допускается сброс признака переполнения
соответствующего банка NAND.</mark>

### 12.4 Обновление MRAM после стирания

<mark>На текущем этапе запись обновлённых счётчиков и служебных данных в MRAM после штатного завершения режима `ERASE`
рассматривается как обязательное действие режима.</mark>

---

## 13. Досрочное завершение по команде

## 13.1 Завершение по `CMD_DUTY`

При получении `CMD_DUTY` в режиме `ERASE` ПО должно:

1. выдать ТС «Квитанция»;
2. инициировать завершение режима;
3. после завершения перейти в `DUTY`;
4. сформировать ТС «Статус».

## 13.2 Завершение по `CMD_SHUTDOWN`

При получении `CMD_SHUTDOWN` в режиме `ERASE` ПО должно:

1. выдать ТС «Квитанция»;
2. инициировать завершение режима;
3. после завершения перейти в `SHUTDOWN`;
4. сформировать ТС «Статус».

## 13.3 Правило досрочного завершения

<mark>На текущем этапе принимается, что если аппаратная реализация не гарантирует безопасного прерывания стирания,
завершение режима `ERASE` по `CMD_DUTY` или `CMD_SHUTDOWN` выполняется после завершения текущего цикла стирания.</mark>

---

## 14. Аварийное завершение режима стирания

### 14.1 Условие

Если в режиме `ERASE` выполняется условие `MaskedAlarm != 0`, ПО должно выполнить переход в `ALARM`.

### 14.2 Действия при аварийном завершении

При аварийном завершении режима `ERASE` ПО должно:

1. прекратить нормальную последовательность стирания;
2. отключить ПЭД;
3. отключить питание `NAND1`;
4. отключить питание `NAND2`;
5. зафиксировать переход в `ALARM`;
6. сформировать ТС «Статус».

### 14.3 Типовые причины аварийного завершения

Для режима `ERASE` особенно существенны:

- отказ питания банка NAND;
- ошибка `QSPI`;
- отказ запуска или выполнения операции стирания;
- выход контролируемых параметров за допустимые пределы.

---

## 15. Обработка разрешённых команд в режиме стирания

## 15.1 `CMD_STATUS_REQ`

При получении `CMD_STATUS_REQ` ПО должно:

1. выдать ТС «Квитанция»;
2. сформировать ТС «Статус»;
3. не изменять режим.

## 15.2 `CMD_DUTY`

При получении `CMD_DUTY` ПО должно:

1. выдать ТС «Квитанция»;
2. инициировать завершение режима;
3. выполнить выход в `DUTY` по принятому правилу завершения;
4. сформировать ТС «Статус».

## 15.3 `CMD_SHUTDOWN`

При получении `CMD_SHUTDOWN` ПО должно:

1. выдать ТС «Квитанция»;
2. инициировать завершение режима;
3. выполнить выход в `SHUTDOWN` по принятому правилу завершения;
4. сформировать ТС «Статус».

## 15.4 Остальные КУ

Остальные КУ:

- отклоняются;
- сопровождаются ТС «Квитанция» с ошибкой;
- не изменяют состояние режима.

---

## 16. Контроль аппаратного состояния в режиме стирания

### 16.1 Контроль статусов питания

В режиме `ERASE` ПО должно контролировать:

- сигнал включения питания активного банка;
- сигнал подтверждения питания активного банка;
- наличие рассогласования между требуемым и фактическим состоянием питания.

### 16.2 Потеря подтверждения питания в процессе стирания

Если в процессе стирания пропадает подтверждение питания активного банка, это должно трактоваться как аварийное событие,
приводящее к аварийному завершению режима.

---
