# Подалгоритм режима вывода данных ПО НА NATALIA

## 1. Назначение документа

Настоящий документ описывает подалгоритм работы штатного ПО в режиме вывода данных.

Документ определяет:

- условия входа в режим вывода данных;
- выбор источника данных и параметров вывода;
- последовательность чтения пакетов НИ из выбранного банка NAND;
- правила передачи данных во внешнюю систему;
- условия штатного завершения;
- условия досрочного завершения по команде;
- условия аварийного завершения;
- действия при выходе из режима.

Документ используется совместно с:

- таблицей состояний и переходов;
- документом обработки команд CAN;
- документом допущений и открытых вопросов;
- документами по остальным режимам работы;
- диаграммами состояний Mermaid.

---

## 2. Назначение режима вывода данных

Режим `DUMP` предназначен для передачи накопленной научной информации из выбранного банка NAND во внешнюю бортовую
систему.

В режиме `DUMP` ПО должно:

- выбрать банк NAND, содержащий данные для вывода;
- при необходимости включить питание выбранного банка;
- определить число пакетов, подлежащих передаче;
- подготовить интерфейс вывода;
- последовательно считывать пакеты НИ из NAND;
- передавать считанные пакеты по USB;
- учитывать число успешно переданных пакетов;
- обновлять служебные данные о ходе вывода;
- завершать режим после передачи требуемого объема данных, по команде или по аварии.

---

## 3. Условия входа в режим вывода данных

### 3.1 Внешнее условие входа

Вход в режим выполняется по КУ:

- `CMD_DUMP`.

### 3.2 Допустимый исходный режим

Вход в режим допускается только из:

- `DUTY`.

### 3.3 Обязательные условия входа

Для успешного запуска режима должны выполняться:

- команда корректна по формату и содержимому;
- выбран корректный банк NAND;
- отсутствует ненулевой `MaskedAlarm`;
- в выбранном банке имеются данные для вывода;
- выбранный банк доступен для чтения;
- интерфейс вывода доступен;
- служба чтения NAND и служба передачи данных готовы к работе.

### 3.4 Причины отказа во входе

Запуск режима должен быть отклонен, если:

- команда получена не из `DUTY`;
- указан недопустимый банк NAND;
- содержимое команды некорректно;
- выбранный банк не содержит данных для вывода;
- невозможно включить или использовать выбранный банк;
- невозможно подготовить интерфейс вывода;
- уже имеется активная авария.

---

## 4. Входные параметры режима вывода данных

Режим вывода данных конфигурируется КУ `CMD_DUMP`.

### 4.1 Параметры команды

Команда задает:

- идентификатор используемого банка NAND:
    - `1` — `NAND1`;
    - `2` — `NAND2`;
- признак состояния питания выбранного банка после завершения вывода:
    - `0` — отключить питание;
    - `1` — оставить питание включенным;
- тип вывода данных:
    - `0` — вывести указанное число пакетов;
    - `1` — вывести все доступные пакеты;
- число пакетов для вывода, если выбран тип `0`.

### 4.2 Интерфейс вывода

В алгоритмической логике режима `DUMP` используется только интерфейс USB.

---

## 5. Внутренние данные режима вывода

### 5.1 Постоянные на операцию

Во время режима `DUMP` ПО должно поддерживать:

- выбранный банк NAND;
- тип вывода;
- требуемое число пакетов для передачи;
- требуемое состояние питания банка после завершения;
- индекс первого пакета текущей операции вывода, если это требуется внутренней логикой.

### 5.2 Динамические данные

ПО должно поддерживать:

- текущее подсостояние режима;
- индекс текущего читаемого пакета;
- число успешно переданных пакетов;
- число пакетов, оставшихся к передаче;
- буфер текущего пакета размером `2048` байт;
- флаг завершения вывода;
- флаг ошибки чтения;
- флаг ошибки передачи;
- флаг ошибки интерфейса USB;
- признак наличия `VBUS`;
- признак необходимости повторной передачи текущего пакета.

### 5.3 Служебные данные

ПО должно поддерживать:

- число накопленных пакетов в выбранном банке NAND;
- счетчик последнего успешно выведенного пакета;
- состояние питания выбранного банка;
- состояние интерфейса вывода.

---

## 6. Аппаратная конфигурация режима вывода

### 6.1 Общие положения

Режим `DUMP` требует использования:

- `CAN1`;
- MRAM;
- одного выбранного банка NAND;
- USB;
- интерфейсов контроля параметров и датчиков.

ПЭД в режиме `DUMP` для выполнения целевой операции не используется.

### 6.2 Банк NAND

В режиме `DUMP` допускается использование только одного активного банка NAND.

Перед началом чтения НИ выбранный банк должен быть включен, подтвержден по `PU_NANDx_PSON` и подключен к портам МК.

Одновременное включение двух банков NAND не допускается.

### 6.3 USB

Перед началом передачи данных по USB должно быть подтверждено наличие `VBUS`, после чего линии USB подключаются к портам
МК и интерфейс переводится в состояние готовности к передаче.

### 6.4 Контроль аппаратного состояния

В режиме `DUMP` ПО должно контролировать:

- питание активного банка NAND;
- наличие `VBUS`;
- корректность работы интерфейса USB;
- контролируемые температуры, напряжения и токи.

---

## 7. Последовательность входа в режим вывода данных

При успешном принятии `CMD_DUMP` ПО должно:

1. выдать ТС «Квитанция» о принятии команды;
2. сохранить параметры команды;
3. выбрать банк NAND;
4. если питание банка выключено, подать сигнал включения питания банка;
5. дождаться подтверждения наличия питания банка по `PU_NANDx_PSON`;
6. подключить сигнальные линии интерфейса выбранного банка NAND к портам;
7. определить число пакетов к передаче:
    - по полю команды, если выбран вывод фиксированного числа пакетов;
    - по числу фактически накопленных пакетов, если выбран вывод всех данных;
8. проверить наличие `VBUS`;
9. подключить линии USB к портам МК;
10. подготовить интерфейс USB к передаче;
11. перевести верхний режим в `DUMP`;
12. сформировать ТС «Статус`;
13. начать цикл вывода пакетов.

Если на этапе подготовки отсутствует `VBUS`, режим `DUMP` не запускается.

---

## 8. Подсостояния режима вывода

Для внутренней логики режим `DUMP` разделяется на подэтапы:

| Подсостояние          | Обозначение         | Назначение                                                |
|-----------------------|---------------------|-----------------------------------------------------------|
| Подготовка входа      | `DUMP_ENTER`        | Выбор банка, включение питания, расчет объема вывода      |
| Подготовка интерфейса | `DUMP_IF_PREP`      | Подготовка USB к передаче                                 |
| Чтение пакета         | `DUMP_READ`         | Чтение очередного пакета из NAND                          |
| Передача пакета       | `DUMP_SEND`         | Передача пакета по USB                                    |
| Проверка завершения   | `DUMP_CHECK`        | Проверка условия продолжения вывода                       |
| Штатное завершение    | `DUMP_FINISH_OK`    | Завершение вывода и выход                                 |
| Завершение по команде | `DUMP_FINISH_CMD`   | Выход по `CMD_DUTY` или `CMD_SHUTDOWN`                    |
| Завершение по аварии  | `DUMP_FINISH_ALARM` | Выход при `MaskedAlarm != 0` или ошибке интерфейса/чтения |

---

## 9. Основные источники событий в режиме вывода

### 9.1 Разрешенные КУ

В режиме `DUMP` разрешены:

- `CMD_SHUTDOWN`;
- `CMD_DUTY`;
- `CMD_STATUS_REQ`.

Остальные КУ запрещены и сопровождаются квитанцией с ошибкой.

### 9.2 КТ

Все КТ в режиме `DUMP` игнорируются.

### 9.3 Внутренние события

В режиме `DUMP` существенны:

- подтверждение включения питания NAND;
- готовность интерфейса USB;
- успешное чтение пакета;
- ошибка чтения пакета;
- успешная передача пакета;
- ошибка передачи пакета;
- исчезновение `VBUS`;
- потеря подтверждения питания активного банка NAND;
- появление `MaskedAlarm != 0`.

---

## 10. Основной цикл режима вывода

### 10.1 Общая логика

После входа в режим `DUMP` ПО должно выполнять повторяющийся цикл:

1. считать очередной пакет НИ из NAND;
2. передать пакет по USB;
3. увеличить число успешно переданных пакетов;
4. обновить служебные данные о ходе вывода;
5. проверить, требуется ли продолжение вывода.

### 10.2 Приоритет обработки событий

В режиме `DUMP` используется следующий логический приоритет:

1. `MaskedAlarm != 0`;
2. `CMD_SHUTDOWN`;
3. ошибка чтения или передачи;
4. завершение текущей передачи;
5. `CMD_DUTY`;
6. `CMD_STATUS_REQ`.

---

## 11. Подготовка источника данных и интерфейса вывода

### 11.1 Подготовка банка NAND

ПО должно:

1. выбрать требуемый банк NAND;
2. обеспечить его питание;
3. подтвердить его питание по `PU_NANDx_PSON`;
4. подключить сигнальные линии интерфейса банка NAND к портам;
5. проверить доступность чтения;
6. определить число доступных пакетов.

### 11.2 Подготовка интерфейса USB

ПО должно:

1. проверить наличие `VBUS`;
2. подключить линии USB к портам;
3. инициализировать службу передачи по USB;
4. перевести интерфейс в состояние готовности к передаче;
5. зафиксировать готовность к началу вывода.

### 11.3 Ошибки подготовки

Ошибками подготовки считаются:

- недоступность банка NAND;
- невозможность чтения;
- отсутствие `VBUS`;
- отказ инициализации интерфейса USB;
- внутренняя ошибка службы передачи.

---

## 12. Этап чтения пакета

### 12.1 Назначение

На этапе `DUMP_READ` ПО должно считать один пакет НИ объемом `2048` байт из выбранного банка NAND.

### 12.2 Действия алгоритма

ПО должно:

1. определить номер текущего пакета;
2. считать пакет в буфер;
3. проверить успешность чтения;
4. передать буфер на этап передачи.

### 12.3 Ошибки этапа чтения

Ошибками этапа чтения считаются:

- отказ чтения;
- ошибка `QSPI`;
- потеря питания банка NAND;
- нарушение целостности считывания, если соответствующий контроль используется.

---

## 13. Этап передачи пакета

### 13.1 Назначение

На этапе `DUMP_SEND` ПО должно передать считанный пакет по USB.

### 13.2 Действия алгоритма

ПО должно:

1. инициировать передачу текущего буфера;
2. дождаться подтверждения успешной передачи либо завершения операции;
3. при успехе увеличить счетчик переданных пакетов;
4. обновить счетчик последнего успешно выведенного пакета;
5. подготовить переход к проверке условия продолжения вывода.

### 13.3 Ошибки этапа передачи

Ошибками этапа передачи считаются:

- тайм-аут передачи;
- отказ интерфейса USB;
- исчезновение `VBUS`;
- иная ошибка передачи, приводящая к невозможности завершить отправку пакета.

### 13.4 Повторная передача пакета

При ошибке передачи одного пакета допускается одна повторная попытка передачи этого же пакета.

При повторной неудаче режим завершается нештатно.

---

## 14. Проверка завершения вывода

### 14.1 Условие завершения при выводе фиксированного числа пакетов

Если выбран режим вывода фиксированного числа пакетов, режим завершается, когда число успешно переданных пакетов
достигает заданного значения.

### 14.2 Условие завершения при выводе всех данных

Если выбран режим вывода всех пакетов, режим завершается, когда переданы все фактически доступные пакеты выбранного
банка NAND.

### 14.3 Действие при незавершенности

Если остаются пакеты, подлежащие передаче, алгоритм должен перейти к чтению следующего пакета.

---

## 15. Штатное завершение режима вывода

### 15.1 Условие завершения

Режим `DUMP` считается завершенным штатно, если:

- передан весь требуемый объем данных;
- не возникло аварии;
- не возникло неустранимой ошибки чтения или передачи.

### 15.2 Действия при штатном завершении

При штатном завершении ПО должно:

1. зафиксировать число успешно переданных пакетов;
2. обновить счетчик последнего выведенного пакета;
3. записать актуальные служебные данные режима в MRAM;
4. оставить питание банка включенным или отключить его по параметру команды;
5. отключить линии USB от портов;
6. перейти в `DUTY`;
7. сформировать ТС «Статус».

---

## 16. Досрочное завершение по команде

### 16.1 Завершение по `CMD_DUTY`

При получении `CMD_DUTY` ПО должно:

1. выдать ТС «Квитанция»;
2. инициировать завершение режима;
3. завершить передачу текущего пакета, если она уже начата;
4. обновить счетчик последнего успешно выведенного пакета в MRAM;
5. после завершения перейти в `DUTY`;
6. сформировать ТС «Статус».

### 16.2 Завершение по `CMD_SHUTDOWN`

При получении `CMD_SHUTDOWN` ПО должно:

1. выдать ТС «Квитанция»;
2. инициировать завершение режима;
3. завершить передачу текущего пакета, если она уже начата;
4. обновить счетчик последнего успешно выведенного пакета в MRAM;
5. после завершения перейти в `SHUTDOWN`;
6. сформировать ТС «Статус`.

### 16.3 Правило досрочного завершения

При отсутствии аварии режим `DUMP` завершается на границе пакетов. Если передача текущего пакета уже начата, алгоритм
должен завершить передачу этого пакета до выхода из режима.

---

## 17. Аварийное завершение режима вывода

### 17.1 Условие

Если в режиме `DUMP` выполняется условие `MaskedAlarm != 0`, ПО должно выполнить переход в `ALARM`.

### 17.2 Действия при аварийном завершении

При аварийном завершении режима `DUMP` ПО должно:

1. прекратить вывод данных;
2. отключить линии USB от портов;
3. отключить ПЭД;
4. отключить питание `NAND1`;
5. отключить питание `NAND2`;
6. зафиксировать переход в `ALARM`;
7. сформировать ТС «Статус`.

### 17.3 Типовые причины аварийного завершения

Для режима `DUMP` особенно существенны:

- отказ чтения из NAND;
- ошибка `QSPI`;
- ошибка передачи по USB;
- исчезновение `VBUS`;
- потеря подтверждения питания активного банка NAND;
- выход контролируемых параметров за допустимые пределы.

---

## 18. Обработка разрешенных команд в режиме вывода

### 18.1 `CMD_STATUS_REQ`

При получении `CMD_STATUS_REQ` ПО должно:

1. выдать ТС «Квитанция»;
2. сформировать ТС «Статус»;
3. не изменять режим.

### 18.2 `CMD_DUTY`

При получении `CMD_DUTY` ПО должно:

1. выдать ТС «Квитанция»;
2. инициировать завершение режима;
3. выполнить выход в `DUTY` по принятому правилу завершения;
4. сформировать ТС «Статус».

### 18.3 `CMD_SHUTDOWN`

При получении `CMD_SHUTDOWN` ПО должно:

1. выдать ТС «Квитанция»;
2. инициировать завершение режима;
3. выполнить выход в `SHUTDOWN` по принятому правилу завершения;
4. сформировать ТС «Статус».

### 18.4 Остальные КУ

Остальные КУ:

- отклоняются;
- сопровождаются ТС «Квитанция» с ошибкой;
- не изменяют состояние режима.

---

## 19. Контроль аппаратного состояния в режиме вывода

### 19.1 Контроль статусов питания

В режиме `DUMP` ПО должно контролировать:

- сигнал включения питания активного банка NAND;
- сигнал подтверждения питания активного банка NAND;
- наличие `VBUS`;
- наличие рассогласования между требуемым и фактическим состоянием аппаратуры.

### 19.2 Потеря подтверждения питания или `VBUS`

Если в процессе вывода:

- пропадает подтверждение питания активного банка NAND;
- пропадает `VBUS`;

это должно трактоваться как нештатное событие режима с формированием соответствующего аварийного признака и последующим
пересчетом `MaskedAlarm`.

### 19.3 Контроль параметров

С периодичностью, заданной общей логикой алгоритма, в режиме `DUMP` должны контролироваться:

- `MC_Temp`;
- `PU_Temp`;
- `PED_Temp`;
- `BD_Temp`;
- `PU_Volt`;
- `PU_Curr`;
- `PED_Volt`;
- `PED_Curr`.

При выходе параметра за допустимый диапазон формируется соответствующий аварийный признак и пересчитывается
`MaskedAlarm`.
