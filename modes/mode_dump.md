# Подалгоритм режима вывода данных ПО НА NATALIA

## 1. Назначение документа

Настоящий документ описывает подалгоритм работы штатного ПО в режиме вывода данных.

Документ определяет:

- условия входа в режим вывода данных;
- выбор источника данных и параметров вывода;
- последовательность чтения пакетов НИ из выбранного банка NAND;
- правила передачи данных во внешнюю систему;
- условия штатного завершения;
- условия досрочного завершения по команде;
- условия аварийного завершения;
- действия при выходе из режима.

Документ используется совместно с:

- таблицей состояний и переходов;
- документом обработки команд CAN;
- документом допущений и открытых вопросов;
- документами по остальным режимам работы;
- диаграммами состояний Mermaid.

---

## 2. Назначение режима вывода данных

Режим `DUMP` предназначен для передачи накопленной научной информации из выбранного банка NAND во внешнюю бортовую
систему.

В режиме `DUMP` ПО должно:

- выбрать банк NAND, содержащий данные для вывода;
- при необходимости включить питание выбранного банка;
- определить число пакетов, подлежащих передаче;
- подготовить интерфейс вывода;
- последовательно считывать пакеты НИ из NAND;
- передавать считанные пакеты по интерфейсу вывода;
- учитывать число успешно переданных пакетов;
- завершать режим после передачи требуемого объёма данных, по команде или по аварии.

---

## 3. Условия входа в режим вывода данных

### 3.1 Внешнее условие входа

Вход в режим выполняется по КУ:

- `CMD_DUMP`

### 3.2 Допустимый исходный режим

Вход в режим допускается только из:

- `DUTY`

### 3.3 Обязательные условия входа

Для успешного запуска режима должны выполняться:

- команда корректна по формату и содержимому;
- выбран корректный банк NAND;
- отсутствует ненулевой `MaskedAlarm`;
- в выбранном банке имеются данные для вывода;
- выбранный банк доступен для чтения;
- интерфейс вывода доступен;
- служба чтения NAND и служба передачи данных готовы к работе.

### 3.4 Причины отказа во входе

Запуск режима должен быть отклонён, если:

- команда получена не из `DUTY`;
- указан недопустимый банк NAND;
- содержимое команды некорректно;
- выбранный банк не содержит данных для вывода;
- невозможно включить или использовать выбранный банк;
- невозможно подготовить интерфейс вывода;
- уже имеется активная авария.

---

## 4. Входные параметры режима вывода данных

Режим вывода данных конфигурируется КУ `CMD_DUMP`.

### 4.1 Параметры команды

Команда задаёт:

- идентификатор используемого банка NAND:
    - `1` — `NAND1`
    - `2` — `NAND2`
- признак состояния питания выбранного банка после завершения вывода:
    - `0` — отключить питание;
    - `1` — оставить питание включённым
- тип вывода данных:
    - `0` — вывести указанное число пакетов;
    - `1` — вывести все доступные пакеты
- число пакетов для вывода, если выбран тип `0`.

### 4.2 Интерфейс вывода

<mark>На текущем этапе режим `DUMP` разрабатывается только для интерфейса USB. Вариант вывода данных через CAN в
алгоритмической логике не используется.</mark>

---

## 5. Внутренние данные режима вывода

## 5.1 Постоянные на операцию

Во время режима `DUMP` ПО должно поддерживать:

- выбранный банк NAND;
- тип вывода;
- требуемое число пакетов для передачи;
- требуемое состояние питания банка после завершения;
- номер первого пакета текущей сессии вывода, если это требуется внутренней логикой.

## 5.2 Динамические данные

ПО должно поддерживать:

- текущее подсостояние режима;
- индекс текущего читаемого пакета;
- число успешно переданных пакетов;
- число пакетов, оставшихся к передаче;
- буфер текущего пакета размером `2048` байт;
- флаг завершения вывода;
- флаг ошибки чтения;
- флаг ошибки передачи;
- флаг ошибки интерфейса USB;
- признак наличия `VBUS`.

## 5.3 Служебные данные

ПО должно поддерживать:

- число накопленных пакетов в выбранном банке NAND;
- счётчик последнего успешно выведенного пакета;
- состояние питания выбранного банка;
- состояние интерфейса вывода.

---

## 6. Последовательность входа в режим вывода данных

При успешном принятии `CMD_DUMP` ПО должно:

1. Выдать ТС «Квитанция» о принятии команды.
2. Сохранить параметры команды.
3. Выбрать банк NAND.
4. Если питание банка выключено, подать сигнал включения питания банка.
5. Дождаться подтверждения наличия питания банка по предусмотренному статусному сигналу.
6. Определить число пакетов к передаче:
    - по полю команды, если выбран вывод фиксированного числа пакетов;
    - по числу фактически накопленных пакетов, если выбран вывод всех данных.
7. Подготовить интерфейс USB.
8. Проверить наличие `VBUS`.
9. Перевести верхний режим в `DUMP`.
10. Сформировать ТС «Статус».
11. Начать цикл вывода пакетов.

<mark>На текущем этапе отсутствие `VBUS` при запуске режима рассматривается как отказ запуска режима `DUMP`, связанный с
интерфейсом USB.</mark>

---

## 7. Подсостояния режима вывода

Для внутренней логики режим `DUMP` разделяется на подэтапы:

| Подсостояние          | Обозначение         | Назначение                                                |
|-----------------------|---------------------|-----------------------------------------------------------|
| Подготовка входа      | `DUMP_ENTER`        | Выбор банка, включение питания, расчёт объёма вывода      |
| Подготовка интерфейса | `DUMP_IF_PREP`      | Подготовка USB к передаче                                 |
| Чтение пакета         | `DUMP_READ`         | Чтение очередного пакета из NAND                          |
| Передача пакета       | `DUMP_SEND`         | Передача пакета по USB                                    |
| Проверка завершения   | `DUMP_CHECK`        | Проверка условия продолжения вывода                       |
| Штатное завершение    | `DUMP_FINISH_OK`    | Завершение вывода и выход                                 |
| Завершение по команде | `DUMP_FINISH_CMD`   | Выход по `CMD_DUTY` или `CMD_SHUTDOWN`                    |
| Завершение по аварии  | `DUMP_FINISH_ALARM` | Выход при `MaskedAlarm != 0` или ошибке интерфейса/чтения |

---

## 8. Основные источники событий в режиме вывода

## 8.1 Разрешённые КУ

В режиме `DUMP` разрешены:

- `CMD_SHUTDOWN`
- `CMD_DUTY`
- `CMD_STATUS_REQ`

Остальные КУ запрещены и сопровождаются квитанцией с ошибкой.

## 8.2 КТ

Все КТ в режиме `DUMP` игнорируются.

## 8.3 Внутренние события

В режиме `DUMP` существенны:

- подтверждение включения питания NAND;
- готовность интерфейса USB;
- успешное чтение пакета;
- ошибка чтения пакета;
- успешная передача пакета;
- ошибка передачи пакета;
- исчезновение `VBUS`;
- появление `MaskedAlarm != 0`.

---

## 9. Основной цикл режима вывода

### 9.1 Общая логика

После входа в режим `DUMP` ПО должно выполнять повторяющийся цикл:

1. считать очередной пакет НИ из NAND;
2. передать пакет по USB;
3. увеличить число успешно переданных пакетов;
4. проверить, требуется ли продолжение вывода.

### 9.2 Приоритет обработки событий

В режиме `DUMP` использовать следующий логический приоритет:

1. `MaskedAlarm != 0`
2. `CMD_SHUTDOWN`
3. ошибка чтения или передачи
4. завершение текущей передачи
5. `CMD_DUTY`
6. `CMD_STATUS_REQ`

---

## 10. Подготовка источника данных и интерфейса вывода

### 10.1 Подготовка банка NAND

ПО должно:

1. выбрать требуемый банк NAND;
2. обеспечить его питание;
3. проверить доступность чтения;
4. определить число доступных пакетов.

### 10.2 Подготовка интерфейса USB

ПО должно:

1. инициализировать службу передачи по USB;
2. проверить наличие `VBUS`;
3. перевести интерфейс в состояние готовности к передаче;
4. зафиксировать готовность к началу вывода.

### 10.3 Ошибки подготовки

Ошибками подготовки считаются:

- недоступность банка NAND;
- невозможность чтения;
- отсутствие `VBUS`;
- отказ инициализации интерфейса USB;
- внутренняя ошибка службы передачи.

<mark>На текущем этапе ошибки подготовки режима `DUMP`, связанные с USB, трактуются как отказ запуска режима без входа в
`DUMP`.</mark>

---

## 11. Этап чтения пакета

### 11.1 Назначение

На этапе `DUMP_READ` ПО должно считать один пакет НИ объёмом `2048` байт из выбранного банка NAND.

### 11.2 Действия алгоритма

ПО должно:

1. определить адрес или номер текущего пакета;
2. считать пакет в буфер;
3. проверить успешность чтения;
4. передать буфер на этап передачи.

### 11.3 Ошибки этапа чтения

Ошибками этапа чтения считаются:

- отказ чтения;
- ошибка `QSPI`;
- потеря питания банка NAND;
- нарушение целостности считывания, если соответствующий контроль используется.

---

## 12. Этап передачи пакета

### 12.1 Назначение

На этапе `DUMP_SEND` ПО должно передать считанный пакет по USB.

### 12.2 Действия алгоритма

ПО должно:

1. инициировать передачу текущего буфера;
2. дождаться подтверждения успешной передачи либо завершения операции;
3. при успехе увеличить счётчик переданных пакетов;
4. перейти к проверке условия продолжения вывода.

### 12.3 Ошибки этапа передачи

Ошибками этапа передачи считаются:

- тайм-аут передачи;
- отказ интерфейса USB;
- исчезновение `VBUS`;
- иная ошибка передачи, приводящая к невозможности завершить отправку пакета.

### 12.4 Повторная передача пакета

<mark>На текущем этапе при ошибке передачи одного пакета допускается одна повторная попытка передачи этого же пакета.
При повторной неудаче режим завершается нештатно.</mark>

---

## 13. Проверка завершения вывода

### 13.1 Условие завершения при выводе фиксированного числа пакетов

Если выбран режим вывода фиксированного числа пакетов, режим завершается, когда:

- число успешно переданных пакетов достигает заданного значения.

### 13.2 Условие завершения при выводе всех данных

Если выбран режим вывода всех пакетов, режим завершается, когда:

- переданы все фактически доступные пакеты выбранного банка NAND.

### 13.3 Действие при незавершённости

Если остаются пакеты, подлежащие передаче, алгоритм должен:

- перейти к чтению следующего пакета.

---

## 14. Штатное завершение режима вывода

### 14.1 Условие завершения

Режим `DUMP` считается завершённым штатно, если:

- передан весь требуемый объём данных;
- не возникло аварии;
- не возникло неустранимой ошибки чтения или передачи.

### 14.2 Действия при штатном завершении

При штатном завершении ПО должно:

1. зафиксировать число успешно переданных пакетов;
2. обновить счётчик последнего выведенного пакета;
3. оставить питание банка включённым или отключить его по параметру команды;
4. перейти в `DUTY`;
5. сформировать ТС «Статус».

### 14.3 Обновление счётчика последнего выведенного пакета

<mark>На текущем этапе счётчик последнего успешно выведенного пакета рассматривается как обязательный служебный параметр
режима `DUMP`.</mark>

---

## 15. Досрочное завершение по команде

## 15.1 Завершение по `CMD_DUTY`

При получении `CMD_DUTY` ПО должно:

1. выдать ТС «Квитанция»;
2. инициировать завершение режима;
3. после завершения перейти в `DUTY`;
4. сформировать ТС «Статус».

## 15.2 Завершение по `CMD_SHUTDOWN`

При получении `CMD_SHUTDOWN` ПО должно:

1. выдать ТС «Квитанция»;
2. инициировать завершение режима;
3. после завершения перейти в `SHUTDOWN`;
4. сформировать ТС «Статус».

## 15.3 Правило досрочного завершения

<mark>На текущем этапе принимается, что при отсутствии аварии режим `DUMP` завершается на границе пакетов. Если передача
текущего пакета уже начата, алгоритм стремится завершить передачу этого пакета до выхода из режима. Окончательное
правило подлежит дополнительному согласованию.</mark>

---

## 16. Аварийное завершение режима вывода

### 16.1 Условие

Если в режиме `DUMP` выполняется условие `MaskedAlarm != 0`, ПО должно выполнить переход в `ALARM`.

### 16.2 Действия при аварийном завершении

При аварийном завершении режима `DUMP` ПО должно:

1. прекратить вывод данных;
2. отключить ПЭД;
3. отключить питание `NAND1`;
4. отключить питание `NAND2`;
5. зафиксировать переход в `ALARM`;
6. сформировать ТС «Статус».

### 16.3 Типовые причины аварийного завершения

Для режима `DUMP` особенно существенны:

- отказ чтения из NAND;
- ошибка `QSPI`;
- ошибка передачи по USB;
- исчезновение `VBUS`;
- выход контролируемых параметров за допустимые пределы.

---

## 17. Обработка разрешённых команд в режиме вывода

## 17.1 `CMD_STATUS_REQ`

При получении `CMD_STATUS_REQ` ПО должно:

1. выдать ТС «Квитанция»;
2. сформировать ТС «Статус»;
3. не изменять режим.

## 17.2 `CMD_DUTY`

При получении `CMD_DUTY` ПО должно:

1. выдать ТС «Квитанция»;
2. инициировать завершение режима;
3. выполнить выход в `DUTY` по принятому правилу завершения;
4. сформировать ТС «Статус».

## 17.3 `CMD_SHUTDOWN`

При получении `CMD_SHUTDOWN` ПО должно:

1. выдать ТС «Квитанция»;
2. инициировать завершение режима;
3. выполнить выход в `SHUTDOWN` по принятому правилу завершения;
4. сформировать ТС «Статус».

## 17.4 Остальные КУ

Остальные КУ:

- отклоняются;
- сопровождаются ТС «Квитанция» с ошибкой;
- не изменяют состояние режима.

---

## 18. Контроль аппаратного состояния в режиме вывода

### 18.1 Контроль статусов питания

В режиме `DUMP` ПО должно контролировать:

- сигнал включения питания активного банка NAND;
- сигнал подтверждения питания активного банка NAND;
- наличие `VBUS`;
- наличие рассогласования между требуемым и фактическим состоянием аппаратуры.

### 18.2 Потеря подтверждения питания или `VBUS`

Если в процессе вывода:

- пропадает подтверждение питания активного банка NAND;
- пропадает `VBUS`;

это должно трактоваться как нештатное событие режима, приводящее к его аварийному завершению.

---
