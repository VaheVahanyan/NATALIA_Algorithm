# Подалгоритм режима наблюдений ПО НА NATALIA

## 1. Назначение документа

Настоящий документ описывает подалгоритм работы штатного ПО в режиме наблюдений.

Документ определяет:
- условия входа в режим наблюдений;
- состав конфигурации режима;
- последовательность действий при входе;
- обработку внутренних событий в режиме наблюдений;
- правила формирования научной информации;
- обработку КУ и КТ в режиме наблюдений;
- условия автоматического и командного выхода из режима;
- действия при авариях и переполнении NAND.

Документ используется совместно с:
- основным описанием алгоритма;
- таблицей состояний и переходов;
- документом обработки команд CAN;
- документом аварийной и защитной логики;
- описанием форматов научной информации.

---

## 2. Назначение режима наблюдений

В режиме наблюдений ПО должно обеспечивать:
- приём событий от ПЭД;
- управление режимом работы ПЭД;
- накопление научной информации;
- формирование форматов НИ;
- запись научной информации в выбранный банк NAND;
- приём КТ от БВС КА и включение их в поток НИ;
- периодическое формирование служебных форматов;
- контроль заполнения банка NAND;
- завершение наблюдений по команде, по переполнению или по аварии.

---

## 3. Условия входа в режим наблюдений

### 3.1 Внешнее условие входа
Вход в режим наблюдений выполняется по КУ:

- `CMD_OBSERVE_START` (`MSG_ID = 0003h`)

### 3.2 Допустимый исходный режим
Вход в режим наблюдений допускается только из:
- `DUTY`

### 3.3 Обязательные условия входа
Для успешного входа должны выполняться следующие условия:
- команда корректна по формату и содержимому;
- выбран корректный банк NAND;
- выбранный банк NAND не отмечен как переполненный;
- отсутствует ненулевой `MaskedAlarm`;
- параметры конфигурации режима наблюдений допустимы;
- конфигурация ПЭД не содержит запрещённых комбинаций;
- выбранный банк NAND может быть подготовлен к записи;
- при необходимости ПЭД может быть включена и переведена в требуемое состояние.

### 3.4 Причины отказа во входе
Вход в режим наблюдений должен быть отклонён, если:
- команда пришла не из `DUTY`;
- выбран некорректный банк NAND;
- выбранный банк NAND переполнен;
- содержимое команды некорректно;
- не удалось подготовить требуемую аппаратную конфигурацию;
- в процессе подготовки возникла авария.

---

## 4. Входные параметры режима наблюдений

Режим наблюдений конфигурируется командой `CMD_OBSERVE_START`.

### 4.1 Аппаратная конфигурация
В составе КУ задаются:
- выбранный банк NAND;
- признак включения/выключения питания ПЭД;
- признак режима пониженного энергопотребления ПЭД;
- признак разрешения регистрации событий в ПЭД;
- конфигурация состояния ПЭД и NAND после автоматического завершения наблюдений по переполнению.

### 4.2 Параметры режима наблюдений
В составе КУ задаются:
- режим формирования формата «События»;
- `Nmax` для формата «События»;
- режим формирования `Спектр-1/Спектр-2`;
- размер гистограммы `Спектр-1`.

### 4.3 Конфигурация триггера
В составе КУ задаётся:
- конфигурация триггерной логики ПЭД.

---

## 5. Внутренние данные режима наблюдений

Во время режима наблюдений ПО должно поддерживать по меньшей мере следующие данные:

### 5.1 Постоянные на сеанс
- идентификатор сеанса наблюдений;
- номер режима наблюдений;
- активный банк NAND;
- активная конфигурация триггера;
- активные параметры режима наблюдений.

### 5.2 Динамические данные
- номер текущего пакета НИ;
- номер текущего формата внутри сеанса;
- состояние текущего собираемого пакета НИ;
- текущие секундные счётчики;
- текущая гистограмма;
- буфер формата «События»;
- флаг заполнения банка NAND;
- флаг необходимости записи формата «Телеметрия»;
- флаг необходимости завершения режима.

### 5.3 Вспомогательные данные
- статус ПЭД;
- последнее приборное время RTC;
- текущие значения телеметрических параметров;
- результаты последних КТ;
- контрольные суммы пакетов/форматов.

---

## 6. Общая последовательность входа в режим наблюдений

### 6.1 Логическая последовательность
При успешном принятии `CMD_OBSERVE_START` ПО должно:

1. Выдать ТС «Квитанция» о принятии команды.
2. Сохранить параметры запуска наблюдений.
3. Выбрать активный банк NAND.
4. Включить питание выбранного банка NAND, если оно выключено.
5. Дождаться подтверждения наличия питания NAND, если это требуется логикой.
6. Подготовить сервис записи научной информации.
7. При необходимости включить питание ПЭД.
8. При необходимости перевести ПЭД в normal/sleep согласно команде.
9. Установить `PED_INHIBIT` согласно требуемой конфигурации.
10. Применить конфигурацию триггера.
11. Инициализировать внутренние буферы режима наблюдений.
12. Инкрементировать идентификатор сеанса наблюдений.
13. Установить номер режима наблюдений в 0.
14. Перейти в верхний режим `OBSERVE`.
15. Сформировать ТС «Статус».

### 6.2 Первый тик после входа
Согласно описанию форматов НИ, после входа в режим наблюдений на последующем тике `RTC 1 Гц`
должен быть сформирован первый формат «Телеметрия», после чего начинается штатное циклическое
формирование остальных форматов. Это поведение надо явно зафиксировать в алгоритме. :contentReference[oaicite:0]{index=0}

---

## 7. Состав подрежимов внутри режима наблюдений

Для внутренней логики полезно разложить режим наблюдений на подэтапы.

| Подсостояние | Обозначение | Назначение |
|---|---|---|
| Подготовка входа | OBS_ENTER | Подготовка NAND, ПЭД и параметров режима |
| Активное наблюдение | OBS_ACTIVE | Основной цикл приёма событий и формирования НИ |
| Завершение по команде | OBS_EXIT_CMD | Корректное завершение по `CMD_DUTY` или `CMD_SHUTDOWN` |
| Завершение по переполнению | OBS_EXIT_FULL | Автоматическое завершение при заполнении NAND |
| Завершение по аварии | OBS_EXIT_ALARM | Немедленное завершение при `MaskedAlarm != 0` |

Примечание:
верхнеуровневое состояние остаётся `OBSERVE`, а эти подэтапы нужны для внутреннего алгоритма
и будущей диаграммы режима.

---

## 8. Основные источники событий в режиме наблюдений

### 8.1 Внешние команды управления
В режиме наблюдений разрешены:
- `CMD_SHUTDOWN`
- `CMD_OBSERVE_CTRL`
- `CMD_TELEM_REQ`
- `CMD_STATUS_REQ`
- `CMD_DUTY`

Остальные КУ запрещены и должны приводить к квитанции с ошибкой. :contentReference[oaicite:1]{index=1}

### 8.2 Внешние команды телеметрии
В режиме наблюдений разрешены:
- `TLM_TIME_SYNC`
- `TLM_ORBIT`
- `TLM_ATTITUDE`
- `TLM_MAGFIELD`

По ним квитанция не выдается, а данные должны записываться в НИ в виде специальных форматов. 

### 8.3 Внутренние события
В режиме наблюдений существенны:
- тик `RTC 1 Гц`;
- триггер `PED_TG`;
- завершение записи пакета/формата;
- переполнение банка NAND;
- изменение статусов ПЭД/NAND;
- появление аварийного признака;
- ошибки записи НИ;
- изменение конфигурации по `CMD_OBSERVE_CTRL`.

---

## 9. Основной цикл режима наблюдений

### 9.1 Общая идея
В активной фазе наблюдений ПО должно одновременно:
- обрабатывать события от ПЭД;
- отслеживать тик 1 Гц;
- формировать и записывать научные форматы;
- принимать допустимые команды CAN;
- принимать КТ и включать их в поток НИ;
- отслеживать переполнение NAND;
- отслеживать аварии.

### 9.2 Приоритеты событий внутри режима
Рекомендуемый логический приоритет:
1. `MaskedAlarm != 0`
2. `CMD_SHUTDOWN`
3. `NAND_FULL`
4. `CMD_DUTY`
5. `CMD_OBSERVE_CTRL`
6. `RTC 1 Гц`
7. `PED_TG`
8. КТ
9. запросы статуса/телеметрии

---

## 10. Обработка тика RTC 1 Гц

### 10.1 Роль события
Сигнал `RTC 1 Гц` является одной из главных опор режима наблюдений.
По этому тику должны синхронизироваться:
- секундные счётчики;
- форматы «Счетчики»;
- форматы «Спектр-1/2»;
- периодические форматы «Телеметрия»;
- завершение временного окна для формата «События».

Интерфейсы STM32 также прямо указывают, что по `RTC_OUT` в МК должно генерироваться прерывание. :contentReference[oaicite:3]{index=3}

### 10.2 Действия на каждом тике
На каждом тике `RTC 1 Гц` в `OBS_ACTIVE` ПО должно:

1. Зафиксировать переход к новому секундному интервалу.
2. Завершить формирование секундных счётчиков за предыдущую секунду.
3. Сформировать формат «Счетчики».
4. Если включён `Спектр-1`, сформировать формат «Спектр-1`.
5. Если включён `Спектр-2`, сформировать формат «Спектр-2`.
6. Если установлен флаг внеочередной телеметрии, сформировать формат «Телеметрия».
7. Если пришёл 20-секундный интервал, сформировать периодический формат «Телеметрия».
8. Сбросить секундные накопители и гистограммы для нового интервала.
9. Продолжить приём событий.

### 10.3 Первый тик после `CMD_OBSERVE_START`
На первом тике после запуска наблюдений должен быть сформирован формат «Телеметрия». :contentReference[oaicite:4]{index=4}

### 10.4 Тик после `CMD_OBSERVE_CTRL`
После изменения режима наблюдений должен быть:
- инкрементирован номер режима наблюдений;
- на следующем тике 1 Гц сформирован формат «Телеметрия». :contentReference[oaicite:5]{index=5}

### 10.5 Тик после `CMD_DUTY`
После команды завершения наблюдений алгоритм должен:
- завершить запись необходимых итоговых форматов;
- сформировать формат «Телеметрия» на следующем тике 1 Гц;
- затем выйти в `DUTY`.  
Это прямо следует из последовательности записи форматов НИ. :contentReference[oaicite:6]{index=6}

## 11. Обработка события PED_TG

### 11.1 Назначение
Сигнал `PED_TG` означает регистрацию события ПЭД и должен инициировать обработку события.

В описании интерфейсов указано, что по `PED_TG` в МК должно генерироваться прерывание. :contentReference[oaicite:7]{index=7}

### 11.2 Общая последовательность обработки события
При получении `PED_TG`, если регистрация разрешена, ПО должно:

1. Зафиксировать факт прихода триггера.
2. Считать необходимые данные события из ПЭД через интерфейс `PED_REG`.
3. Обновить секундные счётчики.
4. Обновить гистограмму спектра.
5. При необходимости добавить событие в буфер формата «События».
6. По завершении обработки сформировать `PED_TGRES`.
7. Освободить тракт для следующего события.

### 11.3 Зависимость от конфигурации
Фактическая запись события в формат «События» зависит от:
- включён ли вообще формат;
- выбранного режима формирования;
- значения `Nmax`;
- условия по `n(AC1) < n(AC1)max`, если выбран соответствующий режим;
- принадлежности к внутреннему/внешнему радиационному поясу, если это влияет на формирование.

---

## 12. Формирование научной информации

### 12.1 Общие положения
Научная информация должна храниться в NAND в виде пакетов по:
- `1024` 16-битных слов;
- `2048` байт на пакет;
- little-endian для 16-битных слов. :contentReference[oaicite:8]{index=8}

### 12.2 Заголовок пакета НИ
Каждый пакет должен содержать:
- 3 маркерных слова;
- идентификатор сеанса наблюдений;
- номер пакета;
- контрольную сумму предыдущего пакета;
- информационное ядро;
- CRC16 текущего пакета. :contentReference[oaicite:9]{index=9}

### 12.3 Форматы НИ, которые могут формироваться в режиме наблюдений
В режиме наблюдений должны использоваться:
- «События»
- «Счетчики»
- «Спектр-1»
- «Спектр-2»
- «Телеметрия»
- «Синхронизация»
- «Орбита»
- «Ориентация»
- «Геомагнитное поле» :contentReference[oaicite:10]{index=10}

### 12.4 Общие правила записи форматов
Для каждого сформированного формата ПО должно:
1. Сформировать заголовок формата.
2. Сформировать полезную нагрузку.
3. Вычислить CRC заголовка и данных.
4. Добавить формат в текущий пакет НИ.
5. Если формат не помещается, завершить текущий пакет и открыть новый.
6. Записать пакет в NAND.
7. Обновить номер пакета и внутренние счётчики.

### 12.5 Неполный последний пакет
Если сеанс наблюдений завершается, а информационное ядро последнего пакета заполнено не полностью,
оно должно быть дополнено словами `AAAAh` до конца пакета. :contentReference[oaicite:11]{index=11}

## 13. Формат «События»

### 13.1 Когда формируется
Формат «События» формируется опционально.

### 13.2 Когда записывается
Он записывается:
- при накоплении `Nmax` событий в текущем секундном интервале;
- или при завершении секундного интервала с числом событий `N <= Nmax`. :contentReference[oaicite:12]{index=12}

### 13.3 Что должен делать алгоритм
ПО должно:
1. вести буфер текущего набора событий;
2. отслеживать секунду наблюдения;
3. завершать формат либо по `Nmax`, либо по смене секунды;
4. после записи очищать буфер и начинать новый набор.

---

## 14. Формат «Счетчики»

### 14.1 Когда формируется
Формат «Счетчики» формируется каждую секунду по сигналу `RTC 1 Гц`. :contentReference[oaicite:13]{index=13}

### 14.2 Что содержит
Должен содержать:
- число сигналов с детектора D;
- число сигналов AC1;
- число сигналов AC2;
- число сигналов триггера;
- мертвое время за секунду. :contentReference[oaicite:14]{index=14}

---

## 15. Форматы «Спектр-1» и «Спектр-2»

### 15.1 Общие правила
Во время наблюдений может формироваться либо:
- `Спектр-1`, либо
- `Спектр-2`,
но не оба одновременно. :contentReference[oaicite:15]{index=15}

### 15.2 Спектр-1
Формируется каждую секунду по тикам `RTC 1 Гц` после «Счетчиков».  
Размер гистограммы задаётся параметром команды запуска/управления наблюдением. :contentReference[oaicite:16]{index=16}

### 15.3 Спектр-2
Формируется каждую секунду после «Счетчиков» в режиме zero suppression.  
Размер гистограммы фиксирован `2048` каналов. :contentReference[oaicite:17]{index=17}

---

## 16. Формат «Телеметрия»

### 16.1 Когда формируется
Формат «Телеметрия» формируется:
- каждые 20 секунд;
- на первом тике после `CMD_OBSERVE_START`;
- на первом тике после `CMD_OBSERVE_CTRL`;
- на первом тике после `CMD_DUTY` перед окончанием записи. :contentReference[oaicite:18]{index=18}

### 16.2 Что должен сделать алгоритм
ПО должно:
1. считать текущие телеметрические параметры;
2. включить статус НА, статус ПЭД, триггерную конфигурацию и параметры режима;
3. сформировать формат;
4. записать его в поток НИ.

---

## 17. Обработка КТ в режиме наблюдений

### 17.1 TLM_TIME_SYNC
При получении КТ «Сверка времени» ПО должно:
1. принять данные бортового времени;
2. сформировать формат «Синхронизация»;
3. записать его в NAND. 

### 17.2 TLM_ORBIT
При получении КТ «Параметры орбиты» ПО должно:
1. принять орбитальные параметры;
2. сформировать формат «Орбита»;
3. записать его в NAND. 

### 17.3 TLM_ATTITUDE
При получении КТ «Параметры ориентации» ПО должно:
1. принять данные ориентации;
2. сформировать формат «Ориентация»;
3. записать его в NAND. 

### 17.4 TLM_MAGFIELD
При получении КТ «Геомагнитное поле» ПО должно:
1. принять вектор геомагнитного поля;
2. сформировать формат «Геомагнитное поле»;
3. записать его в NAND. 

---

## 18. Обработка разрешённых КУ в режиме наблюдений

### 18.1 CMD_OBSERVE_CTRL
При успешной обработке `CMD_OBSERVE_CTRL` ПО должно:
1. выдать квитанцию;
2. обновить конфигурацию ПЭД;
3. обновить параметры наблюдений;
4. инкрементировать номер режима наблюдений;
5. запланировать внеочередной формат «Телеметрия» на следующем тике 1 Гц;
6. остаться в режиме `OBSERVE`. :contentReference[oaicite:23]{index=23}

### 18.2 CMD_TELEM_REQ
При получении `CMD_TELEM_REQ`:
1. выдать квитанцию;
2. сформировать ТС «Телеметрия»;
3. режим не менять.

### 18.3 CMD_STATUS_REQ
При получении `CMD_STATUS_REQ`:
1. выдать квитанцию;
2. сформировать ТС «Статус»;
3. режим не менять.

### 18.4 CMD_DUTY
При получении `CMD_DUTY`:
1. выдать квитанцию;
2. инициировать корректное завершение наблюдений;
3. перейти в подэтап `OBS_EXIT_CMD`;
4. завершить запись НИ;
5. перейти в `DUTY` с формированием ТС «Статус».

### 18.5 CMD_SHUTDOWN
При получении `CMD_SHUTDOWN`:
1. выдать квитанцию;
2. инициировать завершение наблюдений;
3. сохранить нужные служебные данные;
4. перейти в `SHUTDOWN` с формированием ТС «Статус».

---

## 19. Контроль заполнения NAND

### 19.1 Общее правило
При заполнении активного банка NAND режим наблюдений должен завершаться автоматически,
а ПО должно перейти в `DUTY` с установкой признака заполнения банка в статусе. :contentReference[oaicite:24]{index=24}

### 19.2 Что должен делать алгоритм
При обнаружении заполнения активного банка:
1. запретить начало новых операций записи форматов;
2. завершить текущую согласованную запись;
3. установить флаг переполнения банка;
4. привести состояние ПЭД и NAND к конфигурации, заданной в КУ запуска;
5. инициировать выход `OBS_EXIT_FULL`;
6. перейти в `DUTY`;
7. сформировать ТС «Статус`.

### 19.3 Что запрещено
Если банк уже отмечен как заполненный, запуск режима наблюдений в него запрещён. :contentReference[oaicite:25]{index=25}

---

## 20. Выход из режима наблюдений

### 20.1 Варианты выхода
Выход из режима наблюдений возможен:
- по `CMD_DUTY`;
- по `CMD_SHUTDOWN`;
- по переполнению NAND;
- по переходу в `ALARM`.

### 20.2 Общие действия при корректном выходе
При корректном завершении наблюдений ПО должно:
1. завершить текущие форматы;
2. корректно завершить последний пакет НИ;
3. дополнить неполный пакет словами `AAAAh`, если нужно;
4. обновить счётчики накопленных пакетов;
5. при необходимости обновить MRAM;
6. перевести аппаратную конфигурацию в состояние целевого режима выхода;
7. сформировать ТС «Статус`.

---

## 21. Выход по аварии

### 21.1 Условие
Если в режиме наблюдений `MaskedAlarm != 0`, ПО обязано перейти в `ALARM`. :contentReference[oaicite:26]{index=26}

### 21.2 Действия
При аварийном выходе ПО должно:
1. прекратить наблюдение;
2. запретить регистрацию событий в ПЭД;
3. отключить питание ПЭД;
4. отключить оба банка NAND;
5. зафиксировать переход в `ALARM`;
6. сформировать ТС «Статус`.

### 21.3 Открытый вопрос
Требуется отдельно решить, должен ли алгоритм пытаться корректно завершить текущий пакет НИ
при аварийном выходе, или аварийный переход выполняется без завершения текущего пакета.

## 24. Открытые вопросы
