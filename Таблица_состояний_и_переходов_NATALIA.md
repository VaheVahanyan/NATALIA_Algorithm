# Таблица состояний и переходов ПО НА NATALIA

## 1. Назначение документа

Настоящий документ содержит формальное описание автомата состояний штатного ПО
платы управления гамма-спектрометра НА NATALIA.

Документ определяет:
- перечень состояний верхнего уровня;
- перечень событий, влияющих на состояние ПО;
- условия выполнения переходов;
- действия, выполняемые при переходах;
- результирующее состояние после обработки события.

Документ является основой для:
- построения state diagram;
- проверки полноты алгоритма;
- последующей реализации логики на языке C.

---

## 2. Правила использования таблицы переходов

Каждый переход описывается пятёркой:

**(текущее состояние, событие, условие, действие, следующее состояние)**

Где:
- **текущее состояние** — режим, в котором находится ПО до обработки события;
- **событие** — внешняя команда, внутреннее событие или аварийное условие;
- **условие** — дополнительное логическое ограничение;
- **действие** — набор операций, которые должно выполнить ПО;
- **следующее состояние** — режим после завершения обработки события.

Если событие недопустимо в текущем режиме:
- для КУ формируется ТС «Квитанция» с ошибкой;
- для КТ команда игнорируется;
- переход состояния не выполняется.

## 3. Перечень состояний верхнего уровня

| ID | Обозначение | Наименование состояния | Примечание |
|---|---|---|---|
| S0 | INIT | Инициализация после включения питания | Логическое стартовое состояние алгоритма |
| S1 | DUTY | Дежурный режим | Базовый режим работы |
| S2 | ERASE | Режим стирания ППЗУ | Целевой режим |
| S3 | TEST | Режим тестирования ППЗУ | Целевой режим |
| S4 | OBSERVE | Режим наблюдений | Целевой режим |
| S5 | DUMP | Режим вывода данных | Целевой режим |
| S6 | ALARM | Аварийный режим | Режим защитной обработки |
| S7 | SHUTDOWN | Режим выключения | Подготовка к снятию питания |

Примечание:
- состояние `INIT` вводится как алгоритмическое стартовое состояние, даже если в CAN-протоколе
  оно отдельно не кодируется;
- в ТС «Статус» после старта предыдущий режим может указываться как исходное состояние.

## 5. Общие правила приоритета событий

При одновременном возникновении нескольких событий приоритет их обработки принимается следующим:

1. `E_MASKED_ALARM_SET`
2. `E_CMD_SHUTDOWN`
3. завершение длительных операций (`E_ERASE_DONE`, `E_TEST_DONE`, `E_DUMP_DONE`)
4. команды управления
5. команды телеметрии
6. периодические события (`E_RTC_1HZ`)
7. служебные события повторной передачи ТС

Примечание:
точный механизм очередности событий в коде будет определён отдельно,
но логически аварийные события имеют высший приоритет.

## 7. Автоматические действия без смены верхнего состояния

Вне зависимости от текущего режима, если это допускается режимом и не противоречит документации,
ПО должно выполнять ряд фоновых действий.

| № | Состояние | Событие | Действие | Смена состояния |
|---|---|---|---|---|
| A1 | Любое | E_CAN_TX_ERROR | Повторить передачу ТС до 3 раз | Нет |
| A2 | Любое | E_CAN_TX_RETRY_EXHAUSTED | Продолжить выполнение алгоритма без смены режима | Нет |
| A3 | Любое активное состояние | E_STATUS_CHANGED | Автоматически выдать ТС «Статус» | Нет |
| A4 | OBSERVE | E_RTC_1HZ | Формирование обязательных форматов по правилам режима наблюдений | Нет |

## 8. Замечания к детализации

Настоящая таблица описывает только переходы верхнего уровня.

Дополнительной детализации требуют:
- внутренняя логика режима наблюдений;
- последовательность формирования и записи научной информации;
- последовательность операций стирания и тестирования NAND;
- последовательность операций вывода данных;
- логика обработки аварийных признаков;
- логика инициализации при старте ПО.

Эти части описываются в отдельных документах и поддиаграммах.

## 9. Открытые вопросы

В данном разделе фиксируются переходы и условия, требующие уточнения:
- точный состав действий при прерывании длительных операций командой «Дежурный режим»;
- допустимость немедленного прерывания теста/стирания;
- момент фиксации завершения наблюдений;
- правила завершения неполного пакета НИ;
- правила завершения вывода данных при `E_CMD_SHUTDOWN`;
- точные внутренние события от драйверов интерфейсов и периферии.
